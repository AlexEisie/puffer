# puffer-connector-matrix

## Purpose

`puffer-connector-matrix` connects Matrix rooms to Puffer Code sessions. It
supports direct conversations and mentioned group-room conversations.

## Design

The crate uses `matrix-sdk` for Matrix sync and room messaging, while Puffer
conversation state and agent execution live in `puffer-connector-core`.

Matrix configuration covers homeserver URL, username, password, allowed users,
mention behavior, group-key policy, and optional welcome text.

## Logic

Message handling must:

- Sync room events through the Matrix client.
- Ignore messages from the bot account itself.
- Enforce `allowed_users` by Matrix user ID when configured.
- Treat direct rooms as addressable.
- Require a mention in group rooms by default.
- Strip the addressed bot mention from the text before dispatch.
- Build stable conversation IDs from the Matrix room and sender.
- Execute built-in connector commands before agent turns.
- Split outbound text with the Matrix splitter limit.

## Contract

The connector expects password login configuration. End-to-end encrypted room
support is intentionally out of scope for the current connector feature set.

Group-room session behavior must honor the shared `GroupKeyPolicy`.

The Matrix SDK event stream is an input adapter only; the Puffer session store
remains the persisted transcript authority.

## Architecture Details

The Matrix connector wraps Matrix sync and room send APIs around the shared
connector-core runtime. Matrix account login, sync tokens, room classification,
and mention parsing stay local to this crate.

The deterministic handler path should be testable without a live Matrix server:
given normalized sender, room, text, and mention context, it decides ignore,
local command, or agent dispatch.

## Contract Details

- Required config: homeserver URL, username, and password.
- Optional config: allowed Matrix user IDs, mention policy, group-key policy,
  and welcome text.
- Direct-room contract: direct rooms are addressable without mention.
- Group-room contract: mentions are required by default and stripped before
  dispatch.
- Identity contract: allowed users compare against full MXIDs.
- Encryption contract: E2EE rooms are not supported in the current feature set.
- Output contract: replies must be sent to the originating room and split by
  Matrix-safe limits.
