# puffer-connector-discord

## Purpose

`puffer-connector-discord` connects Discord gateway events to Puffer Code
sessions. It lets users talk to Puffer from direct messages or mentioned guild
channels.

## Design

The crate keeps Discord SDK concerns in `connector` and keeps pure message
handling testable. It uses Serenity for gateway access, converts Discord
messages into `puffer-connector-core` inputs, and sends split responses back to
Discord.

Config covers token, allowed users, mention requirements, group-key policy, and
optional welcome text.

## Logic

Message handling must:

- Ignore messages authored by bots, including the Puffer bot itself.
- Enforce `allowed_users` when configured.
- Treat direct messages as always addressable.
- In guild channels, require a bot mention by default.
- Strip `<@bot_id>` or `<@!bot_id>` mention text before dispatch.
- Build conversation keys from channel and user information.
- Run built-in connector commands before agent dispatch.
- Split outbound text with the Discord splitter limit.
- Retry transient send failures with the shared backoff helper.

## Contract

The connector accepts Discord config from the raw connector TOML table. The
required credential is a Discord bot token. `allowed_users` contains Discord
user IDs as unsigned integers.

Guild conversations default to `require_mention = true`. Group session behavior
must honor the shared `GroupKeyPolicy`.

The crate must not store Discord messages as the session source of truth. The
Puffer session store remains authoritative after a message has been dispatched.

## Architecture Details

Discord-specific code is split between gateway/client handling and pure command
handling. The gateway layer extracts message IDs, channel IDs, author IDs,
guild/direct context, mention state, and bot identity. The handler layer makes
the deterministic routing decision and calls connector-core.

The connector must preserve Discord threads/channels only as external
conversation identity. It must not leak Serenity types into connector-core.

## Contract Details

- Required config: bot token.
- Optional config: `allowed_users`, `require_mention`, `group_key_policy`, and
  welcome text.
- Default mention rule: direct messages are accepted; guild messages require a
  bot mention unless configured otherwise.
- Identity contract: allowed users are Discord user IDs; comparisons are numeric.
- Mention contract: accepted guild text should have bot mention syntax removed
  before dispatch.
- Output contract: replies are sent to the originating channel and must respect
  Discord length limits.
- Safety contract: bot-authored messages are ignored to prevent loops.
