# Cross-backend stdout normalization

## Purpose

Keep the local-runner versus gRPC-runner equivalence test focused on stable
tool semantics.

## Design

The cross-backend test still compares each tool result by success, tool ID,
read-state update count, and deterministic stdout content.

Structured stdout may include `durationMs`. That field reflects wall-clock
runtime and can differ between the in-process local runner and the loopback
gRPC runner by a few milliseconds, even when the result is semantically
identical.

## Logic

Before strict stdout comparison, JSON object stdout is parsed and `durationMs`
is removed. Non-JSON stdout and non-object JSON stdout remain byte-for-byte
comparisons.

Workspace paths are still normalized separately before this comparison.

## Contracts

- Cross-backend equivalence must not depend on wall-clock timing fields.
- Tool stdout that is not structured JSON remains a strict string comparison.
- JSON object stdout remains strict after removing known volatile timing.

## Compatibility

This only changes test normalization. It does not alter the gRPC transport,
runner API, or tool output returned to callers.

## Tests

`cargo test -p puffer-runner-grpc --test grpc_e2e cross_backend_equivalence`
should pass even when local and remote `durationMs` values differ.
