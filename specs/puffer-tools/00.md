# puffer-tools

## Purpose

`puffer-tools` defines Puffer tool schemas, built-in tool definitions, typed
tool inputs, tool execution, and adapters from declarative resources to runtime
tool registries.

## Design

The crate separates schema, registry, and execution:

- Tool schema types describe JSON-compatible model-facing input contracts.
- Built-in definitions provide Bash, file, directory, search, and editing
  tools.
- `ToolRegistry` loads declarative tool specs from resources and resolves
  aliases.
- Execution supports built-in handlers, provider-backed tools, runtime tools,
  external command handlers, and shared-library handlers.

The crate does not decide whether a tool is allowed. Permission policy is owned
by `puffer-core`; this crate executes requests it receives.

## Logic

Registry construction must:

- Convert loaded `ToolSpec` resources into typed definitions.
- Skip disabled specs and unsupported approval/enabled-if combinations.
- Preserve aliases.
- Mark MCP-backed and runtime-backed tools.
- Render agent and config descriptions from resources when needed.

Execution logic must:

- Parse JSON input into typed tool inputs when possible.
- Resolve filesystem paths relative to the current working directory.
- Prevent parent traversal for path tools.
- Run Bash through the configured shell with timeout.
- Read files with optional offset/limit.
- Write files by creating parent directories.
- Replace exact old text and support explicit replace-all behavior.
- Move, remove, list, and search paths through typed handlers.
- Dispatch provider/runtime/external/shared-library handlers by tool kind.

## Contract

`ToolDefinition` is the model-facing contract: name, description, input schema,
metadata, policy, display hints, and kind.

`ToolInput` and `TypedToolInput` are the execution input contract for built-in
tools. Invalid or missing fields must produce tool errors, not panics.

`ToolExecutionResult` is the output contract. It must indicate success, carry
tool output text/data, and preserve enough error text for the agent and UI to
show useful feedback.

External command handlers receive JSON on stdin and environment variables
including `PUFFER_TOOL_ID`, `PUFFER_TOOL_NAME`, and `PUFFER_TOOL_HANDLER`.

## Architecture Details

`puffer-tools` is the execution library below `puffer-core` permissions. It
knows how tools are described and run, but not whether a model is allowed to use
them in a given session. That separation lets Anthropic/OpenAI adapters share
schemas while `puffer-core` enforces policy.

Execution backends are intentionally pluggable:

- Built-in typed Rust handlers.
- Provider-backed pseudo-tools that must be handled by runtime/provider code.
- Runtime-local tools such as agent, skill, browser, notebook, MCP, sleep, and
  workflow helpers.
- External process handlers.
- Shared-library ABI handlers.

## Contract Details

- Schema contract: tool schemas must be valid JSON-schema-like shapes accepted
  by provider adapters.
- Alias contract: registry lookup must resolve declared aliases to canonical
  tool definitions.
- Input contract: JSON input is parsed into typed structs before built-in
  execution; parse failures produce tool errors.
- Filesystem contract: path tools resolve under cwd and reject parent traversal
  unless the higher runtime explicitly uses a broader permission mode.
- Bash contract: command execution has timeout behavior and returns captured
  stdout/stderr/status in tool output.
- Edit contract: replacement tools require exact old text unless creating a new
  file or explicitly replacing all occurrences.
- Runtime contract: provider/runtime tools are marked so callers can intercept
  them instead of trying to run them as built-ins.
- ABI contract: shared-library handlers must provide the expected symbol and
  free returned strings through the expected free function.
