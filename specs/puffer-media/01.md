# puffer-media 01: 声明式媒体生成引擎(生图 + 生视频)

- 日期:2026-06-11
- 状态:设计已确认,待写实现计划
- 约束:**不考虑向后兼容**;仅优化长期收益、稳定、性能;**防止过度设计**

## 1. 背景与问题

当前生图/生视频每接一个 provider 就手写一个"全栈适配器"(建 body + 认证 + 轮询 + 解析 + 下载)。
对照三家真实请求方式(byteplus / relaydance / kling),差异其实集中在 4 点:**认证、请求体编码、生命周期、响应解析**,其余约 80% 重复。代码核对发现的具体问题:

- 媒体层用 `reqwest::blocking`,而宿主运行时是 tokio 异步 —— 阻抗失配。
- 每个在途渲染占一条阻塞线程,整段渲染时长 `thread::sleep` 轮询。
- `Client::new()` 散落各适配器,无连接池复用。
- 零幂等:重试/网络抖动可能重复提交、重复扣费。
- 崩溃恢复靠"下次用户点生成时顺手 reclaim",非主动。
- 派发分裂成两处 `match adapter.as_str()`(`video.rs` / `runtime.rs`)+ `MediaExecutionKind` 枚举;加一个 provider 要碰 6 处文件。
- image 路径同步、video 路径异步,双脑。

参考主流方案:fal(统一队列、sync=退化 async、per-model 输出 schema、幂等 request_id、自动重试);LiteLLM(provider config + transform_request/transform_response 钩子 + 统一错误映射);Vercel AI SDK(统一 provider spec + 能力元数据)。

## 2. 目标

1. **易扩展**:加一个"已知协议"的 provider/model = 改 YAML,零 Rust。
2. **稳定**:崩溃可恢复、重试不重复扣费、错误按类处理。
3. **性能**:连接池复用、在途任务不占阻塞线程整段时长。

## 3. 关键决策(已确认)

| 维度 | 决策 | 理由 |
|---|---|---|
| 并发模型 | **保持 blocking + 全局共享 Client + 一条后台 reconciler 线程** | 拿到约 90% 性能/稳定收益,避免 async 全迁移的回归面。防止过度设计。 |
| 声明式深度 | **中等**:execution 块放「provider+kind」级(沿用现有结构),富化字段映射/状态词表/结果指针;**不引**跨 provider 协议预设继承层 | 现阶段 3-4 家,继承/合并层是过度设计且是 bug 面;spec 结构预留扩展门,等真重复再抽。 |
| 稳定性范围 | **纳入**:提交即落库 task_id + 后台 reconciler 主动恢复;错误三分类驱动重试;幂等 id(支持的厂商用 `external_task_id`,不支持则本地查重降级)。**不纳入**:Webhook。 | Webhook 桌面端无公网端点,YAGNI。其余是真稳定收益。 |
| 认证 | 新增 `StoredCredential::AccessKeySecret{access_key, secret_key}` + `AuthMode::AccessKeySecret` | 无兼容包袱,JWT 一等公民,优于 `ak\|sk` 塞 ApiKey。 |
| 同步生图 | 作为 `lifecycle: inline`(提交即终态),并入同一引擎 | 消除 image/video 双脑。 |
| 交付次序 | **strangler**:先迁移现有各家(golden 对齐)→ Kling 放最后作为可扩展性证明 | 先用已知正确行为证明引擎,再加新功能;每步即时删旧,不留双系统长窗口。 |

被否定的两个极端:全量异步 + 跨 provider 协议预设 DSL(过度设计);只加 Kling 不动结构(留债)。

## 4. 架构总览

```
工具层  Video/ImageGeneration tool                                   (不变)
  │
选择层  resolver(保留):逻辑模型+轴 → ResolvedRequest
  │     { kind, model_id, params(by request_field), prompt, image_refs, client_request_id }
  │
资产层  asset:// → url 或 base64(按 spec.image_refs.encoding)        (新,提交前一步)
  │
引擎层  MediaEngine::submit(spec, resolved) → MediaJob(已落 task_id 即返回)
  │       ├ AuthStrategy   Bearer | JwtHs256(ak/sk, 按 exp 缓存)
  │       ├ RequestCodec   按 spec.request 拼 body(+ 可选 quirk)
  │       └ ResponseCodec  按 spec.response(JSON 指针+status_map)解析
  │
对账层  Reconciler(1 条后台 worker 线程 + 信号量限并发)
  │       扫非终态 job → poll → 终态下载/解码 → 工件(video 加 poster)
  │       退避带抖动;错误三分类
  │
基座层  MediaJob 状态机/sidecar(保留) + 全局共享 reqwest::blocking::Client(连接池)
```

控制流只有一条:**submit 只负责"建任务+落 task_id+返回";等待全交给一个 reconciler。**
`inline` 生图在 submit 内直接拿到结果,reconciler 不介入。

## 5. 模块边界

每个单元须满足:能单测(注入式 transport)、不读内部即知用途、改内部不破坏调用方。

| 模块(`crates/puffer-media/src/media/`) | 做什么 | 依赖 |
|---|---|---|
| `spec.rs` | catalog execution 块 → `ExecutionSpec`,带 schema 校验 | serde |
| `auth.rs` | `AuthStrategy::{Bearer, JwtHs256}` 加认证头;JWT 现签+按 exp 缓存 | hmac/sha2/base64 |
| `request_codec.rs` | 通用 body 构建(model_field/prompt/image_refs/constants/点号嵌套);调 quirk | spec |
| `response_codec.rs` | JSON 指针取 task_id/status/result;status_map;错误分类 | spec |
| `engine.rs` | `submit()`:鉴权→编码→发→解析→落库→返回 job | 上四者 + http + jobs |
| `reconciler.rs` | 后台线程:批量 poll 非终态 job、退避抖动、终态→output | engine/jobs/output |
| `output.rs` | 下载或 b64 解码 → 工件;video 追加 poster | http + jobs + video_poster |
| `http.rs` | 全局共享 Client、超时、URL 安全、secret 脱敏 | reqwest |
| `quirks/` | 极少数 provider 怪癖(byteplus 数字 duration、generate_audio 默认关) | — |

接口草图(签名意图,非最终):

```rust
trait AuthStrategy { fn apply(&self, req: RequestBuilder) -> Result<RequestBuilder>; }
trait RequestCodec { fn submit_body(&self, r: &ResolvedRequest) -> Result<Value>; }
trait ResponseCodec {
    fn parse_submit(&self, v: &Value) -> Result<SubmitOutcome>; // task_id | inline 结果
    fn parse_poll(&self,   v: &Value) -> Result<PollState>;     // status + result_urls + error_class
}
enum Lifecycle { SubmitPoll, Inline }
trait ProviderQuirks {
    fn after_build_body(&self, _body: &mut Value, _r: &ResolvedRequest) {}
    fn override_status(&self, _raw: &str) -> Option<MediaJobStatus> { None }
}
```

## 6. 数据模型

### execution 块(provider+kind 级,该 provider 所有同类 model 共享)

以 Kling 视频为例:

```yaml
media:
  video:
    execution:
      protocol: submit_poll_task            # submit_poll_task | inline
      auth: { mode: jwt_hs256 }
      base_url: https://api-singapore.klingai.com
      endpoint: /v1/videos/text2video
      endpoint_with_image: /v1/videos/image2video
      request:
        model_field: model_name             # 默认 "model"
        prompt: { mode: field, field: prompt }        # field | content_array
        image_refs:                                   # unsupported | content_image_url | named_fields
          mode: named_fields
          fields: [image, image_tail]
          encoding: url                                # url | base64
        idempotency_field: external_task_id
        constants: {}                                  # 固定字段,如 generate_audio:false
        # 参数位置复用每个轴的 request_field,点号支持嵌套 "metadata.ratio"
      response:
        task_id_pointer: /data/task_id
        status_pointer: /data/task_status
        status_map: { submitted: queued, processing: running, succeed: succeeded, failed: failed }
        result_pointer: /data/task_result/videos      # 数组
        result_url_field: url
        error_pointer: /data/task_status_msg
        retryable_http: [429, 500, 502, 503, 504]
    models:                                  # model 只写 axes + variants(沿用现状)
      - id: kling-v2-1
        operations: [generate]
        axes: [ ... mode/duration/aspect_ratio ... ]
        variants: { model_name: kling-v2-1 }
```

三家 + 生图同时落到这张表(验证够用):

| 字段 | byteplus 视频 | relaydance 视频 | kling 视频 | kling 生图 | openai 生图 |
|---|---|---|---|---|---|
| protocol | submit_poll_task | submit_poll_task | submit_poll_task | submit_poll_task | inline |
| auth | bearer | bearer | jwt_hs256 | jwt_hs256 | bearer |
| model_field | model | model | model_name | model_name | model |
| prompt.mode | content_array | field | field | field | field |
| image_refs.mode | content_image_url | unsupported | named_fields | named_fields(image) | — |
| constants | generate_audio:false | n:1 | — | — | — |
| 参数嵌套 | flat | metadata.* | flat | flat | flat |
| result_pointer | task→video_url | 同 | task_result/videos[].url | task_result/images[].url | data[].url/b64 |

### MediaJob 扩展(现有结构上加)

`client_request_id`、`provider_task_id`(已有)、`next_poll_at_ms`、`attempt`、`error_class`。

## 7. 数据流

- **submit_poll_task**:resolver → 资产解析 → engine.submit(鉴权/编码/POST)→ ResponseCodec 取 task_id → **落库(queued)即返回** → reconciler 后续 poll → 终态 → output 下载 → 工件。
- **inline**(同步生图):engine.submit 直接拿结果 → output → 工件(job 一建即 succeeded),reconciler 不参与。

## 8. 错误处理

- 三分类:`Retryable`(网络 / `retryable_http` / 429)→ 退避+抖动重试(上限 attempt);`Terminal`(4xx / 校验 / 内容审核)→ 立即 failed,不再轮询;`Auth`(401 / JWT 过期)→ 标记待重认证。
- 未知 **task_status**(非 HTTP 层)沿用现有保守哲学:当 running 继续轮询。
- 所有错误串过 `redact_secrets`;reconciler 单 job 失败不拖垮其它。
- 退避带抖动(避免同秒齐发)。

## 9. 测试策略

- **契约 fixture**:每 provider 一组 golden 请求体 + 轮询响应 json,回放过 engine 断言(沿用现有 `Fake*Transport` 注入法)。
- 迁移 byteplus/relaydance 时用其现有 fixture 锁行为对齐(回归基线)。
- **加 provider = 加 fixture + execution 块,引擎代码不动。**

## 10. 删除 / 保留 / 不做

- **删**:`byteplus_video.rs` / `relaydance_video.rs` / `replicate_video.rs` / `images_json.rs` / `minimax_image.rs` / `chat_image_output.rs` 的全栈逻辑;`MediaExecutionKind` 枚举;`video.rs`+`runtime.rs` 双 `match adapter`;`reject_unsupported_video_image_references`;散落 `Client::new()`;"启动顺手 reclaim"。
- **保留**:resolver 轴/变体层;MediaJob/工件/sidecar;video_poster;URL 安全/脱敏;注入式测试法。
- **不做**:跨 provider 协议预设继承;Webhook;async 全迁移。

## 11. 范围与分期(每步独立可编译可测)

1. 基座:`http.rs` 共享 client + reconciler 线程骨架 + MediaJob 扩展。
2. 引擎四件套:spec / auth / request_codec / response_codec + engine.submit。
3. 迁 byteplus + relaydance 视频到引擎(golden 对齐),删旧。
4. 迁生图(openai / minimax / chat)为 inline,删旧。
5. 接 Kling:视频(t2v/i2v)+ 生图 = execution 块 + JwtHs256 +(可能零 quirk)。
6. 退役 `MediaExecutionKind` 与双派发。

## 12. 风险与边界

- 共享 client 与 reconciler 引入需与 daemon 任务编排对齐;reconciler 线程生命周期(启动/优雅停止)需明确。
- 幂等依赖各家支持 `external_task_id`:Kling 明确支持;byteplus/relaydance 需逐一确认等价键,缺失则降级"本地查重 + 提交前查重"。
- JSON 指针需支持数组(`task_result/videos[].url`);spec 反序列化必须 schema 校验,配契约测试防静默错配。
- Kling i2v 仅支持公网 `https://` 或裸 base64;`asset://` 在资产层解析为 url/base64,否则明确报错。
- 坚决不做纯 DSL:怪癖走 `quirks/` 钩子。
