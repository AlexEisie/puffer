# puffer-provider-registry

## Purpose

`puffer-provider-registry` defines provider and model descriptors, stores
provider credentials, imports external credentials, and performs optional model
discovery.

## Design

The crate is the typed provider catalog for Puffer. It separates:

- Static provider/model descriptors loaded from resources.
- Runtime registry ordering and lookup.
- Credential storage and secure OAuth token material handling.
- Model compatibility metadata for Anthropic Messages, OpenAI Responses,
  OpenAI Codex Responses, Azure OpenAI Responses, and OpenAI Chat Completions.
- Discovery clients that merge remote model lists into existing providers.

Credential metadata is persisted in JSON, while OAuth token secrets are stored
through the secure OAuth backend when available.

## Logic

Registry logic must:

- Preserve provider insertion order.
- Register providers with source metadata.
- Apply OpenAI base URL, header, and query-parameter overrides from config.
- Resolve models by provider and model ID.
- Merge discovered models without deleting static descriptors.

Auth-store logic must:

- Treat missing auth files as empty stores.
- Load format version 1 and legacy shapes.
- Hydrate OAuth token secrets from secure storage.
- Save API keys inline and OAuth metadata separately from token secrets.
- Remove stale OAuth secret entries when credentials are removed.

Discovery logic must:

- Use a disk cache with a one-hour freshness window.
- Discover only eligible providers, such as providers with auth or local URLs.
- Add auth headers appropriate to the provider family.
- Parse OpenAI, Anthropic, and Ollama-shaped model-list responses.
- Preserve caller-provided query parameters and custom headers.

## Contract

`ProviderDescriptor` is the provider contract. It includes ID, display name,
base URL, default API, auth modes, headers, query params, discovery config, and
models.

`ModelDescriptor` is the model contract. It includes ID, display name, provider,
API family, context limits, output limits, reasoning support, modality support,
cost, and compatibility flags.

`StoredCredential` is either an API key or OAuth credential. OAuth credentials
must preserve account, organization, email, plan, rate, scopes, roles, expiry,
and token metadata without exposing token secrets unnecessarily.

Discovery must be additive and best-effort. A failed discovery request must not
erase bundled provider metadata.

## Architecture Details

The registry is the authoritative provider/model catalog for the process. It
combines declarative resources, runtime config overrides, credential state, and
remote discovery. Transport crates consume descriptors; they do not discover or
persist providers themselves.

Credential architecture is split:

- Provider descriptors are non-secret metadata.
- API keys are stored in the auth JSON format.
- OAuth metadata is stored in auth JSON.
- OAuth token secrets are stored through the secure OAuth backend when possible.
- Import probes read external tool config without making those files canonical.

## Contract Details

- Provider ID contract: IDs are stable lookup keys and must remain unique.
- Model ID contract: models are scoped to providers; provider-qualified selectors
  should resolve without ambiguity.
- Source contract: every registered provider has `ProviderSource` provenance.
- Override contract: OpenAI base URL/header/query overrides mutate only the
  built-in OpenAI provider unless caller explicitly registers another provider.
- Auth format contract: format version 1 and legacy auth files must load; saves
  should normalize to current format.
- Discovery contract: remote models are merged additively; static models and
  model compatibility metadata are not removed on discovery failure.
- Cache contract: discovery cache freshness is time-bounded and should not hide
  explicit provider refresh errors forever.
- Secret contract: OAuth token material should not be emitted in provider lists,
  model lists, diagnostics, or debug output.
