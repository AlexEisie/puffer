# Telegram delivery cursor

- The Telegram user subscriber now persists `delivery-cursor.json` in the
  account state directory. The file records the highest message id emitted per
  chat and is independent from Telegram's MTProto update cursor.
- On the first authorized run without a delivery cursor, the subscriber records
  the latest visible dialog messages as the baseline so newly-created workflows
  do not backfill an arbitrary account history.
- On later restarts, the subscriber scans recent dialogs before entering the
  live update loop and emits messages newer than the persisted per-chat cursor.
  This replays messages received while Puffer was offline even if Telegram's
  session state has already advanced.
- Every emitted live or catch-up message advances the per-chat cursor only
  after the ndjson event is written to stdout. Live updates are deduped against
  the same cursor to avoid duplicate workflow actions after catch-up.
- Catch-up is bounded to 500 dialogs and 5000 messages per chat to avoid an
  accidental unbounded account-history crawl while still covering normal
  offline/restart windows.
