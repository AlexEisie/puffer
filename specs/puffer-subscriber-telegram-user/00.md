# puffer-subscriber-telegram-user

## Purpose

`puffer-subscriber-telegram-user` is the hidden Telegram user-account subscriber
process. It logs into Telegram through MTProto, emits incoming personal-account
messages as NDJSON events, and supports control-driven login.

## Design

The process uses `grammers-client` and runs as a stdio subscriber. Stdout is the
event protocol, stderr is diagnostics. Persistent state lives under
`PUFFER_SKILL_STATE_DIR`.

State includes the MTProto session file and persisted API credentials. The
process can use explicit credentials, persisted credentials, environment
variables, or the Telegram Desktop default API ID/hash.

## Logic

Startup must:

- Resolve state dir and topic from environment, defaulting to `./state` and
  `telegram-user`.
- Resolve API ID/hash from explicit command/config, persisted state,
  `PUFFER_TELEGRAM_API_ID`/`PUFFER_TELEGRAM_API_HASH`, or defaults.
- Attempt to resume an authenticated MTProto session.
- Emit `ready` when authenticated.
- Emit `login_required` and wait for login commands when unauthenticated.

Login flow must:

1. Accept `TelegramLoginStart` with phone and optional credentials.
2. Request a login code and emit `login_awaiting_code`.
3. Accept `TelegramLoginSubmitCode`.
4. Complete login or emit `login_awaiting_password`.
5. Accept `TelegramLoginSubmitPassword` when needed.
6. Persist session and credentials after successful login.

Message handling must emit events with chat, sender, message ID, date, text,
outgoing flag, dedup key, and channel/message kind metadata.

## Contract

The process reads `SubscriberCommand` JSON lines from stdin and writes `Event`
JSON lines to stdout. Login progress and errors are represented as control
events, not human-formatted stdout.

This crate is for personal Telegram accounts. Telegram bot behavior belongs to
`puffer-connector-telegram`.

Telegram session files and credentials are stored in the state directory. The
caller is responsible for protecting that directory.

## Architecture Details

The telegram-user subscriber is a supervised MTProto client process. It is
separate from the Telegram bot connector because it authenticates as a personal
Telegram account and receives a different event stream.

The process owns login state, MTProto session persistence, update handling, and
optional outbound send commands. It does not own subscription routing or Puffer
agent execution.

## Contract Details

- Environment contract: `PUFFER_SKILL_STATE_DIR` controls session/credential
  files and `PUFFER_SKILL_TOPIC` controls event topic.
- Credential contract: API ID/hash resolution order is explicit credentials,
  persisted credentials, environment variables, then Telegram Desktop defaults.
- Login contract: login progress is reported through control events, never
  through prose stdout.
- Session contract: authenticated MTProto sessions are persisted and reused on
  restart.
- Event contract: each message event includes chat, sender, message ID, date,
  outgoing flag, kind, text, payload metadata, and dedup key.
- Output contract: stdout is strict NDJSON events; stderr is diagnostics only.
- Safety contract: personal-account secrets and session files remain confined to
  the declared state directory.
