# puffer-resources

## Purpose

`puffer-resources` loads and merges declarative Puffer resources: providers,
prompts, tools, skills, plugins, MCP servers, LSP servers, IDE metadata,
mascots, agents, and hooks.

## Design

The crate treats resources as layered data with provenance. Built-in embedded
resources load first, then optional filesystem built-ins, user resources, and
workspace resources. Later layers replace earlier items by ID while preserving
diagnostics that describe replacements and load errors.

Core resource types include:

- `ToolSpec`
- `PromptTemplate`
- `AgentSpec`
- `HookSpec`
- `SkillSpec`
- `PluginSpec`
- `McpServerSpec`
- `LspServerSpec`
- `MascotSpec`
- `IdeSpec`
- provider packs that produce provider descriptors

## Logic

Loading logic must:

1. Read embedded resources.
2. Overlay filesystem built-ins when configured.
3. Overlay user resources from the user Puffer directory.
4. Overlay workspace resources from `.puffer/resources`.
5. Normalize names and IDs where required.
6. Preserve `SourceInfo` for every loaded item.
7. Collect diagnostics for invalid files, disabled manifests, and replacements.

Prompt lookup must support base prompt IDs plus provider/model variants.
`prompt_for` falls back from provider-and-model to model, provider, and base
prompt. Rendering must support parent chaining, variable defaults, variable
substitution, and cycle detection.

Plugin loading must merge plugin-provided agents and expose plugin MCP/LSP
helpers without losing the plugin source.

Skill loading must parse `SKILL.md` frontmatter, normalize skill names, and
respect `user_invocable` and `disable_model_invocation`.

## Contract

`LoadedResources` is the read-only runtime resource snapshot. It must contain
the loaded item collections and diagnostics for the caller to display.

Resource provenance is part of the contract. Callers must be able to tell
whether an item came from built-in, user, or workspace resources.

Layering is last-writer-wins by resource ID, but replacement diagnostics must
make overrides auditable.

Resource parsing must be deterministic and filesystem-order independent after
normalization/sorting. Missing optional resource directories are not errors.

## Architecture Details

Resources are Puffer's declarative behavior layer. The crate turns a filesystem
and embedded bundle into a typed, provenance-preserving snapshot. Other crates
consume that snapshot but should not parse resource files themselves.

Resource layers are ordered by authority:

1. Embedded built-ins.
2. Filesystem built-ins.
3. User resources.
4. Workspace resources.

Plugins are resource containers. Plugin-provided agents, skills, MCP servers,
and LSP servers remain traceable to their plugin and source layer.

## Contract Details

- Identity contract: every resource kind has a stable ID or normalized name used
  for replacement and lookup.
- Provenance contract: `LoadedItem<T>` must retain source kind, path, and plugin
  context where applicable.
- Diagnostics contract: invalid files, disabled resources, and replacements are
  reported without aborting the whole load unless the caller cannot continue.
- Prompt contract: provider/model variants and chained prompts must resolve
  deterministically and reject cycles.
- Tool contract: declarative tools provide schema, handler, policy, metadata,
  and display hints for `puffer-tools` and provider adapters.
- Skill contract: `SKILL.md` frontmatter controls model/user invocation and must
  not be ignored when exposing skill commands or model tools.
- MCP/LSP contract: canonical directories and legacy names are normalized so old
  user resources still load.
- Exclusion contract: resource loading must not execute plugin code.
