# Connected Image Provider Resources

## Summary

Provider resources should describe executable image-generation capabilities for
every currently supported connected provider. They should not advertise
providers that only support chat, image input, or unverified router defaults.

This update expands provider media resources beyond OpenAI while keeping
resource metadata declarative and bounded.

## Resource Shape

`media.image` should support:

```yaml
media:
  image:
    adapter: images_json
    connection:
      kind: auth_store
    discovery:
      adapter: static
    execution:
      path: /images/generations
      response: openai_data_array
    models:
      - id: concrete-model-id
        display_name: Concrete Model
        operations:
          - generate
        parameters:
          - name: size
            values: ["1024x1024"]
            default: "1024x1024"
            request_field: size
```

The schema should remain intentionally small. Initial parameters are select-only
string enums with defaults and optional request-field aliases. Resource metadata
may name a bounded response parser kind. It must not become an arbitrary HTTP
templating language.

## Static Descriptors

Static descriptors should be added for providers with stable, documented image
model ids and request shapes:

- OpenAI GPT Image models with `openai_images`;
- BytePlus ModelArk Seedream models with `images_json`;
- Zhipu GLM/CogView image models with `images_json`;
- xAI Grok Imagine image models with `images_json`;
- MiniMax and MiniMax CN `image-01` with `minimax_image`.

Provider ids and auth ids should be canonical. The BytePlus resource should use
`byteplus` instead of `bytepluses`.

## Dynamic Descriptors

Router and gateway providers should use trusted discovery only when the
provider exposes structured image-output metadata:

- OpenRouter can use model metadata filtered to image output;
- Vercel AI Gateway can use image-generation model metadata or explicit static
  image-only models;
- WorldRouter remains hidden until it exposes equivalent official metadata.

Discovered models must still map to an implemented adapter. A discovered model
without an executable adapter is not an available capability.

## Local Providers

Local providers should not advertise image capability from resources alone.
Local image generation is deferred behind an explicit local-provider slice:

- vLLM may later appear when a vLLM-Omni image endpoint responds as expected;
- Ollama may later appear when experimental image generation is available;
- llama.cpp, LM Studio, and MiniCPM5 remain hidden for image generation.

No default settings load should probe local providers.

## Validation

Resource loading should fail on:

- empty concrete image model ids;
- `auto`, wildcard, or regex-like media model ids;
- empty required parameter value lists;
- media adapters unknown to the current binary;
- response parser kinds unknown to the current binary;
- static media descriptors for providers marked image-input-only or text-only.
- non-select parameter declarations until a tested adapter requires another
  parameter kind.

Validation must not require network access.
