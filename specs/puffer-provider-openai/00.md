# puffer-provider-openai

## Purpose

`puffer-provider-openai` implements OpenAI-compatible authentication, request
building, response parsing, and usage helpers for Puffer Code.

## Design

The crate exposes a curated API from the crate root. It separates:

- API key and OAuth bearer auth state.
- Codex-compatible OAuth URL, exchange, and refresh helpers.
- Typed Responses API and Chat Completions request structures.
- HTTP request construction with ordered headers, custom headers, and query
  parameters.
- Response parsing for visible text, reasoning, tool calls, and usage.

The crate is transport construction and parsing only. It does not own the agent
loop or session state.

## Logic

OAuth logic must:

- Generate PKCE verifier/challenge pairs.
- Build the OpenAI Codex OAuth authorization URL with state and originator.
- Parse pasted codes, callback URLs, query strings, or `code#state` values.
- Exchange codes for tokens.
- Refresh tokens through the configured or default token endpoint.

Request-building logic must:

- Normalize base URLs so `/v1` is not duplicated.
- Select `/v1/responses` for Responses API requests and
  `/v1/chat/completions` for Chat Completions requests.
- Add content type, authorization, originator, session ID, client request ID,
  ChatGPT account, and caller-provided headers.
- Add SSE `Accept` behavior for streaming Responses requests.
- Append configured query parameters.

Response parsing must preserve reasoning summaries and encrypted reasoning
content where present, extract `reasoning_content`/`reasoning`, support
`<think>...</think>` fallback extraction, remove hidden thinking from visible
assistant text, and return typed tool-call data.

## Contract

The public auth contract is `OpenAIAuth::None`, `ApiKey`, or `OAuthBearer`.
Callers must choose the auth mode based on provider-registry credentials.

The request contract is a fully built JSON POST request containing URL, ordered
headers, and body. The caller owns actual HTTP execution.

The parser contract is tolerant of OpenAI-compatible servers but typed enough
to preserve assistant text, reasoning, tool calls, usage, and structured output
signals.

Usage helpers may query OpenAI usage and cost endpoints only when the base URL
supports them. Unsupported local or compatible servers must return explicit
usage errors, not fabricated totals.

## Architecture Details

The crate is a typed HTTP request/response helper, not a client runtime. It
does not execute agent loops, manage sessions, decide tool permissions, or own
credential persistence. Callers provide auth, model/API compatibility data, and
request bodies; the crate returns built requests and parsed responses.

OpenAI-compatible paths are separated so official OpenAI, Codex OAuth, Azure,
local proxies, and Chat Completions compatibility can share structures without
collapsing their different headers and paths.

## Contract Details

- OAuth URL contract: generated URLs must include PKCE challenge, state,
  redirect URI, client ID, scopes, and originator-compatible parameters.
- OAuth exchange contract: parsed callback input may be a raw code, full URL,
  query string, or `code#state` value.
- Header contract: authorization, originator, session, client request ID,
  account, and custom headers must be deterministic and caller-inspectable.
- Path contract: `/v1/responses` and `/v1/chat/completions` are built according
  to selected API family; callers may override compatible base URLs.
- Body contract: typed request structures preserve tools, tool choice,
  structured output, reasoning, stream flag, and prior response/threading data.
- Parser contract: partial-compatible payloads should parse useful assistant
  text/tool/usage data without requiring every OpenAI field.
- Error contract: unsupported usage endpoints, malformed OAuth input, and failed
  token refresh/exchange produce typed errors.
