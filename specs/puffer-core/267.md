# puffer-core Update 245: Exact Image Model Runtime

## Summary

Puffer image generation must be driven by exact executable media model
capabilities. It must not use provider defaults, provider `auto` routes, chat
model availability, input image support, or model-name guesses.

The complete target is descriptor-driven and provider-agnostic:

```text
provider_id + model_id + operation=image.generate + execution_adapter
```

Every registered provider is scanned. A provider appears in image generation
settings only when it is connected, has a concrete image model descriptor or
trusted executable discovery result, and has an implemented adapter that can
execute that exact model. No fallback model is allowed. If the selected model
is unavailable, generation fails before any provider request is sent.

This is not a provider whitelist and not a first-version-only design. The
long-term extensibility comes from adding small API-shape adapters and exact
model descriptors, not from adding provider-specific desktop UI or a generic
request-template language.

## Current Fit And Gaps

The repository already has a media runtime boundary under
`crates/puffer-core/runtime/media`:

- `MediaGenerationService` persists jobs and artifacts.
- `MediaJob` and `MediaJobStatus` normalize job state.
- `OpenAIImageRequest` models the OpenAI Images request body.
- Replicate video support already demonstrates async media job handling.
- The `ImageGeneration` tool writes generated images as artifacts.

The current user-visible gap is integration, not a lack of media structs:

- CLI daemon `list_media_capabilities` only advertises hard-coded
  `openai/gpt-image-1`.
- CLI daemon `generate_media` validates against that hard-coded list and then
  returns a fake queued image job instead of executing an adapter.
- Tauri preview backend duplicates the same OpenAI-only capability cache.
- The core `ImageGeneration` workflow is still hard-wired to the OpenAI API
  key and `https://api.openai.com/v1/images/generations`.
- `MediaSettingsModal` already consumes `listMediaCapabilities(kind)` and
  filters `status === "available"`, so missing BytePlus, WorldRouter, and
  Zhipu entries are primarily a backend capability-resolution issue.

The optimized plan therefore reuses the existing runtime and adds exact-model
capability resolution, instead of replacing it with a larger media platform.

## Design Corrections

The re-check found these scope corrections:

- Keep `ModelDescriptor.input` as input modality only. It must not imply image
  output generation.
- Keep provider chat discovery separate from media discovery. A `/models` list
  without structured image-output metadata does not make a model executable.
- Keep the existing job/artifact storage layout and status vocabulary. Do not
  migrate them only to rename states.
- Do not predeclare adapter enum variants in code before the adapter exists.
  Planned adapter shapes can be documented, but runtime enum values should
  reflect implemented behavior.
- Connect desktop, CLI daemon, and the agent `ImageGeneration` tool to the same
  resolver and adapter path so the UI cannot show models that tools cannot use.
- Validate saved image settings against the selected capability. UI fallbacks
  must not allow unsupported sizes, qualities, or output formats to be sent.

## Provider Registry Contract

`ProviderDescriptor` gains an optional `media` field in
`crates/puffer-provider-registry/src/model.rs`. Existing provider YAMLs continue
to parse because the field is optional.

```rust
struct ProviderDescriptor {
    // existing fields...
    #[serde(default)]
    media: Option<ProviderMediaDescriptor>,
}

struct ProviderMediaDescriptor {
    image: Option<ImageMediaDescriptor>,
}

struct ImageMediaDescriptor {
    discovery: Option<MediaDiscoveryDescriptor>,
    execution: Option<MediaExecutionDescriptor>,
    models: Vec<MediaModelDescriptor>,
}

struct MediaDiscoveryDescriptor {
    adapter: MediaDiscoveryKind,
    path: Option<String>,
    query: BTreeMap<String, String>,
}

struct MediaExecutionDescriptor {
    adapter: MediaExecutionKind,
    path: String,
}

struct MediaModelDescriptor {
    id: String,
    display_name: Option<String>,
    operations: Vec<MediaOperation>,
    parameters: MediaImageParameters,
}

struct MediaImageParameters {
    size: Vec<String>,
    quality: Vec<String>,
    output_format: Vec<String>,
}
```

Initial code enum values should stay as small as the implemented behavior:

```rust
enum MediaDiscoveryKind {
    Static,
}

enum MediaExecutionKind {
    OpenAiImages,
}
```

When an adapter is actually implemented and tested, add the matching enum then.
Likely future shapes are:

- `RouterModelModalities` discovery for routers/gateways with structured image
  modality metadata.
- `OpenAiResponsesImageTool` execution for Responses `image_generation` tool
  output.
- `OpenAiChatImageOutput` execution for chat-completions routers that document
  exact image-output content parts.

Example descriptor:

```yaml
media:
  image:
    discovery:
      adapter: static
    execution:
      adapter: openai_images
      path: /v1/images/generations
    models:
      - id: gpt-image-1
        display_name: GPT Image 1
        operations: ["generate"]
        parameters:
          size: ["1024x1024", "1024x1536", "1536x1024"]
          quality: ["auto", "low", "medium", "high"]
          output_format: ["png", "jpeg", "webp"]
```

Rules:

- `models[].id` must be a concrete provider model id.
- `auto`, empty ids, wildcard ids, and regex ids are invalid image models.
- `execution` must be present and implemented for a model to become available.
- Dynamic discovery may add exact candidates only when the adapter can
  interpret structured media capability metadata.
- YAML must not support regex matching, scripts, prompt probes, web scraping,
  or arbitrary request templates.
- Provider headers, query params, base URL, and auth continue to come from the
  existing `ProviderDescriptor` and `AuthStore`.

## Capability Resolution

The resolver input is:

```text
ProviderRegistry + AuthStore + media kind + operation + cached discovery
```

Default resolution, used by desktop modals and generation:

```text
for provider in registered providers:
  if provider is not connected and provider is not explicit auth-free:
    continue
  if provider.media.image is missing:
    continue
  if provider.media.image.execution is missing:
    continue
  if execution adapter is not implemented:
    continue
  candidates = static concrete models + cached executable discovery models
  for candidate in candidates:
    if candidate.id is empty or candidate.id == "auto":
      continue
    if candidate does not include operation=image.generate:
      continue
    if candidate parameters are invalid for the adapter:
      continue
    emit available capability
```

Verbose resolution, used by diagnostics, may also report skipped and unknown
providers with reasons. Verbose diagnostics are separate from selectable
capabilities so the desktop modal does not need disabled speculative rows.

Capability DTOs should keep the current desktop shape unless a UI need appears:

```rust
struct MediaCapability {
    provider_id: String,
    model_id: String,
    kind: MediaKind,
    operations: Vec<String>,
    supports_async: bool,
    supports_streaming: bool,
    parameter_values: serde_json::Value,
    status: String,
    source: String,
    reason: Option<String>,
    checked_at_ms: u64,
}
```

Only `status == "available"` can be saved or executed.

## Execution Adapters

Adapters are grouped by API shape, not provider name.

### `openai_images`

This is the first required adapter. It executes OpenAI-compatible image
generation:

```text
POST {provider.base_url}{media.image.execution.path}
```

The request always includes the selected exact model id:

```json
{
  "model": "selected-model-id",
  "prompt": "...",
  "size": "...",
  "quality": "...",
  "output_format": "...",
  "n": 1
}
```

The adapter must:

- use provider auth, headers, query params, and base URL from the existing
  provider registry path;
- validate `size`, `quality`, and `output_format` against the selected
  capability before sending;
- parse `data[].b64_json` and `data[].url`;
- download URL outputs and persist local artifacts before marking the job
  succeeded;
- redact secrets in request errors;
- fail with `image generation response did not contain an image` when no
  output image is returned.

Existing OpenAI image request code should be generalized into this adapter
rather than duplicated in daemon and tool paths.

### Future exact adapters

The following are valid target shapes but should not be added to code until a
provider descriptor and tests need them:

- `openai_responses_image_tool`: selected exact Responses model with the
  `image_generation` tool; extract returned image bytes or URLs.
- `openai_chat_image_output`: selected exact chat-completions router/gateway
  model that documents image output content; parse the provider's exact image
  content shape.
- native provider adapters, such as Gemini, fal, Replicate, or ComfyUI, when
  the project intentionally supports those request and lifecycle shapes.

Router and gateway providers must not be exposed from discovery alone. They
become selectable only after both structured image-capability discovery and an
execution adapter are implemented.

## Provider Coverage

Every current provider is scanned, but only executable exact image models are
exposed.

| Provider | Runtime treatment |
| --- | --- |
| `openai` | Available through bundled exact image models using `openai_images` once credentials are present. Responses image-tool support is a later adapter, not required for the Images path. |
| `bytepluses` | Available only when bundled or user media descriptors declare concrete BytePlus image model ids and parameters for `openai_images`. Chat discovery does not imply image output. |
| `zhipu` | Same as BytePlus: concrete image model descriptor plus implemented adapter required. Existing GLM chat models do not appear as image models. |
| `openrouter` | Not available from `auto` or generic model discovery. Requires structured image-output metadata plus `openai_chat_image_output` or another verified adapter. |
| `worldrouter` | Same router treatment as OpenRouter. The current `auto` model is excluded. |
| `vercel-ai-gateway` | Same router/gateway treatment as OpenRouter. |
| `xai`, `groq`, `cerebras` | Available only if a provider or user resource declares concrete image models and an implemented adapter supports that API shape. |
| `kimi-openai` | Available only with concrete image models and an implemented OpenAI-compatible image adapter path. |
| `kimi-coding`, `kimi-*` non-OpenAI variants | Skipped unless a future exact image-output adapter and concrete models are declared. |
| `anthropic` | Skipped for image generation unless Anthropic adds a documented exact image-output API shape and Puffer implements it. Image input support is irrelevant. |
| `minimax`, `minimax-cn`, `minimax*` | Skipped until a MiniMax image-generation adapter and concrete model descriptors are implemented. |
| `ollama`, `vllm`, `llama-cpp`, `lmstudio`, `minicpm5` | Auth-free does not mean image-capable. They appear only with explicit user or workspace media descriptors and a supported adapter. |

The resolver must not infer capability from provider name, model id substrings,
chat API family, or input modality.

## Media Jobs And Artifacts

Use the existing `MediaGenerationService`, `MediaJob`, and `MediaArtifact`
layout unless implementation reveals a concrete incompatibility:

```text
.puffer/media/jobs/<job-id>.json
.puffer/media/artifacts/<artifact-id>/<filename>
.puffer/media/artifact-sidecars/<artifact-id>.json
```

Keep the existing status vocabulary:

```text
queued
running
succeeded
failed
canceled
```

Do not add `streaming`, `canceling`, `expired`, webhook states, or a database
job store for this work. Synchronous image adapters can create a job, move it
to `running`, persist artifacts, and finish as `succeeded` or `failed` in one
request. Async adapters can reuse `queued` and `running` when they are actually
implemented.

Artifact metadata records provider id, model id, adapter kind, prompt, revised
prompt when available, MIME type, local path, optional remote source URL, byte
count, and created timestamp. Provider output bytes must not be stored in
transcripts or long-lived UI state.

## Desktop, Daemon, And Tool Contract

`list_media_capabilities` returns exact available capabilities by default:

```json
{
  "capabilities": [
    {
      "providerId": "openai",
      "modelId": "gpt-image-1",
      "kind": "image",
      "operations": ["generate"],
      "status": "available"
    }
  ]
}
```

Optional verbose calls may include diagnostics:

```json
{
  "capabilities": [],
  "diagnostics": [
    {
      "providerId": "worldrouter",
      "state": "skipped",
      "reason": "image models discovered but no image execution adapter is configured"
    }
  ]
}
```

Desktop behavior:

- The modal continues to show only available capabilities.
- It must not display `Auto`, provider defaults, or unknown candidates as
  selectable image models.
- It may preselect the first available capability locally for convenience, but
  persisted config and execution still require a concrete saved `providerId`
  and `modelId`.
- Parameter controls should prefer selected capability values. If capability
  values exist, the UI should not merge in unsupported global fallback values.
- The backend remains the final validator for selected model and parameters.
- No provider-specific desktop branches are needed.

Daemon behavior:

- Replace OpenAI-only `list_media_capabilities` with the shared resolver.
- Replace fake queued `generate_media` with exact-model validation and adapter
  execution.
- Return a `succeeded` image job for synchronous image adapters.
- Return clear validation errors before any provider request is sent.

Tauri preview backend:

- Remove duplicated OpenAI-only media capability cache, or make it delegate to
  the same resolver used by the daemon path.
- Do not maintain a second provider-specific capability source.

Agent tool behavior:

- The `ImageGeneration` tool must use the same media config, resolver,
  adapter, job, and artifact path as daemon `generate_media`.
- It must stop hard-coding OpenAI API key lookup and the OpenAI endpoint.

Expected `generate_media` errors:

- `image media provider/model is not configured`
- `selected image model unavailable: <provider>/<model>`
- `image media adapter unavailable for <adapter>`
- `image generation parameter unsupported: <name>=<value>`
- `image generation response did not contain an image`
- provider-specific request errors with redacted secrets

## Performance And Stability

- Default modal loads use static descriptors and cached discovery only.
- Dynamic discovery is cached by provider id, base URL, auth identity, media
  descriptor hash, adapter kind, and resource version.
- Discovery failures do not block unrelated providers.
- Base64 image data is decoded at the backend boundary and written to disk
  promptly.
- URL outputs are downloaded before completion so generated artifacts survive
  provider URL expiry.
- Exact selected models are never replaced by fallback models.

## Explicit Non-Goals

- No provider `auto` image model.
- No fallback from one exact model to another model.
- No model-name heuristics.
- No web-page scraping.
- No prompt probe on modal open.
- No arbitrary YAML request-template language.
- No provider-specific desktop UI branches.
- No new media database.
- No full async media platform until a concrete adapter needs it.

## Execution Plan

1. Provider registry schema
   - Add optional `ProviderDescriptor.media` and media descriptor types.
   - Add serde parse tests for old provider YAMLs, valid image descriptors,
     invalid `auto` image ids, missing execution, and invalid parameter lists.

2. Core capability resolver
   - Add a resolver module under existing `crates/puffer-core/runtime/media`.
   - Reuse current `MediaCapability` DTO shape where practical.
   - Resolve available exact image capabilities from registry, auth, descriptor,
     implemented adapter registry, and cached discovery.
   - Add tests for connected, missing-auth, auth-free, missing-media,
     missing-adapter, `auto`, stale selection, and parameter validation cases.

3. `openai_images` adapter
   - Generalize existing OpenAI image request code to use provider base URL,
     descriptor path, auth, headers, and query params.
   - Parse both `b64_json` and URL image outputs.
   - Persist artifacts through existing `MediaGenerationService`.
   - Cover response parsing and artifact writing with unit tests and mocked
     HTTP tests.

4. Provider resources
   - Add bundled media descriptors only for providers with verified concrete
     image model ids and an implemented adapter.
   - Start with OpenAI Images and add BytePlus/Zhipu only when exact model ids,
     paths, and parameter sets are verified.
   - Keep OpenRouter, WorldRouter, and Vercel AI Gateway out of selectable
     capabilities until router/gateway image-output execution is implemented.

5. CLI daemon integration
   - Replace daemon `list_media_capabilities` hard-coding with the resolver.
   - Replace fake queued `generate_media` with exact validation, adapter
     execution, job persistence, and artifact persistence.
   - Preserve current desktop DTO field names.

6. Desktop and Tauri cleanup
   - Remove or delegate Tauri's duplicated OpenAI-only capability cache.
   - Keep `MediaSettingsModal` provider/model filtering, but prevent unsupported
     fallback parameter options from being saved for a selected capability.
   - Add desktop tests for provider/model dropdown behavior, missing saved
     selection, and exact parameter option rendering.

7. Agent `ImageGeneration` unification
   - Make the tool call the shared resolver and adapter path.
   - Preserve output-path handling while recording artifacts through the same
     job/artifact service.
   - Add tests that a non-OpenAI exact provider can execute when its descriptor
     and adapter are available.

8. Final verification
   - Connected providers without `media.image` do not appear.
   - Providers with `media.image` but missing auth do not appear.
   - `openai`, `bytepluses`, and `zhipu` appear only when concrete image models
     are declared and credentials are available.
   - `openrouter`, `worldrouter`, and `vercel-ai-gateway` do not appear from
     `auto` or generic chat discovery.
   - `modelId=auto` never appears as a selectable image model.
   - `generate_media` refuses missing, stale, unknown, or unavailable model ids.
   - Generated images are saved as local artifacts and can be reloaded after a
     desktop refresh.

## Self-Review Notes

- Scope is limited to exact image generation and shared resolver integration.
- Existing media runtime state names and storage paths are preserved.
- The plan handles desktop, daemon, Tauri fallback, and agent tool paths.
- Router/gateway support is included as an extension point, but not exposed
  until an exact execution adapter exists.
- No provider auto route, fallback model, heuristic detection, or arbitrary
  request templating remains in the design.
