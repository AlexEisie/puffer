# Interactive Canvas State

## Design

Canvas now has a typed feedback channel for user interaction. The `Canvas` tool
continues to render declarative component trees into self-contained HTML, but
interactive nodes (`toggle`, `singleSelect`, `multiSelect`, `barSelect`,
`slider`, and `textInput`) are initialized into a sibling state file under
`.puffer/canvas/<canvasId>.state.json`.

Each interactive node must carry a stable `id`; that id becomes the key under
`values`. The new `CanvasState` workflow tool reads either a specific
`canvasId` or the latest updated state file for the current session.

## Contracts

- Canvas ids must start with `canvas-` and contain only ASCII alphanumeric,
  dash, or underscore characters before they are used in paths.
- State reads and writes are session-scoped; a state file whose `sessionId` does
  not match the active session is rejected.
- Browser-originated patches update only the `values` object and append a small
  patch event with changed keys.
- No Canvas state path is accepted from the model or browser; paths are derived
  from the validated id and workspace cwd.

## Compatibility

Static Canvas output is unchanged except for a returned `canvasId` and
`statePath`. Existing `finding` actions now include current Canvas values in
their generated follow-up message when any interactive values exist. CLI mode
still renders and writes the initial state file; live browser updates require
the daemon bridge.
