# Unified browser permission backend

## Purpose

Consolidate the Browser permission rewrite into one `puffer-core` update spec.
This change makes Browser permissions target-aware, session-scope-aware, and
entrypoint-independent across:

- Browser tool calls
- Browser-shaped `Bash` / `PowerShell` calls
- daemon-backed current-tab context hydration
- structured approval payloads shown by UIs

## Design

Browser permission code currently lives behind dedicated modules:

- `permissions/browser_action.rs`
- `permissions/browser_target.rs`
- `permissions/browser_grants.rs`
- `permissions/browser_policy.rs`
- `permissions/browser_evaluator.rs`

In this update, `permissions.rs` stays limited to top-level dispatcher and
Browser-bearing call detection work.

The unified Browser backend has four major pieces:

1. Shared Browser intent normalization
   - Browser tool calls and simple `puffer browser ...` shell commands reduce
     to the same Browser permission payload.
   - Stable Browser action sets are:
     - `Inspect`
     - `Navigate`
     - `Interact`
     - `Evaluate`

2. Structured Browser target/context modeling
   - `BrowserPermissionContext`
   - `BrowserTarget`
   - `BrowserTargetClass`

3. Typed Browser session grants
   - `allow_once`
   - `allow_origin_session`
   - `allow_domain_session`
   - `allow_action_on_origin_session`
   - `allow_tab_session`
   - `cross_session_access`

4. Deterministic Browser policy evaluation
   - hard deny
   - typed grant reuse
   - explicit allow
   - default ask

## Logic

### Browser-bearing detection

A Browser-bearing call is any invocation that can be normalized into the shared
Browser permission payload:

- the `Browser` tool
- `Bash` / `PowerShell` calls that reduce to one simple `puffer browser ...`
  command

Compound or otherwise unstable shell commands that mention `puffer browser`
must not silently fall back to ordinary shell approval semantics. In the
current implementation they return `Ask` with the reason:

`ambiguous browser shell command requires explicit approval`

### Shared Browser action mapping

The shared Browser intent layer maps concrete actions into stable permission
action sets:

- `Inspect`
  - `list`
  - `snapshot`
  - `screenshot`
- `Navigate`
  - `open`
  - `new`
  - `focus`
  - `close`
  - `quit`
  - `exit`
  - `navigate`
  - `reload`
  - `back`
  - `forward`
- `Interact`
  - `focus_ref`
  - `click`
  - `dblclick`
  - `hover`
  - `type`
  - `insertText`
  - `fill`
  - `select`
  - `upload`
  - `check`
  - `uncheck`
  - `press`
  - `keydown`
  - `keyup`
  - `scroll`
  - `scrollIntoView`
- `Evaluate`
  - `evaluate`
  - `eval`

### Structured Browser target/context

Browser permission evaluation now works on one structured target per call.

`BrowserTarget` carries:

- raw URL string
- scheme
- origin
- host
- registrable domain
- port
- target class

`BrowserPermissionContext` carries:

- action
- target
- tab id
- root session id
- cross-session status

Initial target classes are:

- `LocalDev`
- `WorkspaceFile`
- `NonWorkspaceFile`
- `DataUrl`
- `OpenWeb`

`workspace_file` versus `non_workspace_file` is determined from canonicalized
file paths against canonicalized workspace roots (`cwd + working_dirs`).

`about:blank` is explicit but is not promoted to a sixth target class.

Registrable-domain derivation uses `psl`, not an in-tree heuristic table.

### URL and tab-context hydration

Browser actions without an explicit URL hydrate target context before
permission evaluation:

1. normalize the Browser-bearing payload
2. resolve the root Browser session id
3. query the daemon `browser_current_tab` RPC
4. fill missing `tabId`
5. fill missing real `url` when the daemon reports `available`
6. if no real URL can be recovered, fall back to the remembered URL cache when
   present

The daemon-side current-tab snapshot is read-only and returns:

- `status`
- `tabId`
- `url`
- `origin`
- `host`
- `port`
- `title`

Supported statuses are:

- `no_active_tab`
- `empty_url`
- `about_blank`
- `available`

### Typed Browser grants

Browser approvals no longer map back into a synthetic whole-tool Browser allow.
Grant reuse is evaluated against the full `BrowserPermissionContext`, so:

- origin
- registrable domain
- action category
- tab id
- root session id
- cross-session status

all constrain reuse.

Suggested Browser scope selection remains backend-only in the current design.
Frontends do not choose origin/domain/action/tab scopes directly.

### Browser-first precedence

Permission evaluation order is:

1. request-scoped tool filter / static disabled checks
2. Browser-bearing detection
3. Browser evaluator
4. generic allow-all / tool override / per-tool policy fallback

Within the Browser evaluator, order is:

1. action presence / action-set classification
2. hard deny rules from the `[browser]` section
3. matching session-scoped Browser grant
4. target context availability
5. cross-session baseline check
6. explicit allow rules from the `[browser]` section
7. default `ask`

That means:

- Browser-bearing shell calls must not bypass Browser evaluation through
  `allow_all_tools`
- Browser-bearing shell calls must not bypass Browser evaluation through a
  generic `bash` / `powershell` session override
- only non-Browser shell calls may use generic shell-grant short-circuits

### Browser policy source

Workspace Browser policy is currently loaded from the `[browser]` section in
`.puffer/permissions.toml`.

The generic `[tools]` section no longer participates in Browser decisions.
`permissions.tools.browser` is filtered out at load time and does not affect
Browser evaluation in this design.

### Approval payloads and persistence

Browser approval prompts expose one structured Browser payload for both Browser
tool calls and Browser-bearing shell routes:

- `source`
- `action_set`
- `url`
- `origin`
- `host`
- `target_class`
- `tab_id`
- `is_cross_session`

Approval responses remain generic:

- `allow_once`
- `allow_session`
- `allow_all_session`
- `deny`

But Browser persistence stays Browser-first:

- approving a Browser-bearing shell call writes Browser scoped grants, not a
  generic shell tool override
- approving a Browser tool call writes the same Browser scoped grants as the
  equivalent Browser-bearing shell call
- Browser-bearing `allow_all_session` is intentionally treated the same way as
  `allow_session`; it does not widen into a whole-session all-tools grant
- remembered Browser target URLs are shared across Browser tool and shell
  Browser routes

## Contracts

- In this update, Browser permission code should stay out of `permissions.rs`
  except for dispatcher wiring and Browser-bearing detection.
- Browser action-set mapping is defined in one place in `puffer-core`.
- Browser tool calls and simple `puffer browser ...` shell commands must
  normalize to the same Browser action set and permission context.
- Compound or ambiguous `puffer browser ...` shell commands must not silently
  fall back to generic shell allow logic.
- Browser target classification currently exposes the five initial target
  classes listed above.
- Browser permission context must include action, target, tab id, root session
  id, and cross-session status.
- Browser grants remain typed and narrower than a coarse whole-tool Browser
  allow.
- Browser grants must be reusable across Browser tool and Browser-bearing shell
  routes when action set, target, tab, and root session match.
- Browser grant scope selection remains internal to `puffer-core`; prompt UIs
  may display Browser context but must not become origin/domain/action/tab
  selectors.
- Browser session approvals must not route through the session-wide
  `allow_all_tools` switch.
- A Browser-bearing `allow_all_session` response must not silently widen into a
  generic all-tools session grant.
- In the current policy model, workspace Browser policy is loaded from the
  `[browser]` section in `.puffer/permissions.toml`.
- In that same model, generic workspace tool entries for Browser do not affect
  Browser evaluation.
- `BrowserTarget.registrable_domain` remains optional and must use PSL
  semantics for normal hostnames.
- The ambiguity reason string is currently:
  `ambiguous browser shell command requires explicit approval`

## Tests

Coverage should assert:

- Browser target classification for `local_dev`, `workspace_file`,
  `non_workspace_file`, `data_url`, and `open_web`
- `about:blank` behavior is explicit, does not create a sixth target class,
  and falls back to `ask` when no real target can be hydrated
- representative PSL-sensitive cases such as
  `docs.example.co.uk -> example.co.uk`,
  `a.b.example.uk.com -> example.uk.com`,
  and `foo.github.io -> foo.github.io`
- direct Browser actions map to the expected action sets
- shell `puffer browser ...` commands map to the same action sets
- Browser tool and shell Browser commands can produce identical permission
  contexts
- simple `puffer browser ...` shell commands normalize into Browser intents
- compound `puffer browser ... && ...` commands do not normalize into a single
  Browser payload
- ambiguous Browser shell commands return `Ask` with the exact ambiguity reason
- unrelated compound shell commands keep existing allow/ask behavior
- Browser typed session grants cover `allow_once`,
  `allow_origin_session`, `allow_domain_session`,
  `allow_action_on_origin_session`, `allow_tab_session`, and
  `cross_session_access`
- origin/domain/action/tab mismatches do not reuse Browser session grants
- the `[browser]` section can hard-deny by target class, origin, and domain
- the `[browser]` section can explicitly allow by target class, origin, and
  domain
- generic `permissions.tools.browser` does not affect Browser decisions
- Browser-bearing shell calls still ask when Browser evaluator asks, even if
  the session is in `allow_all_tools`
- shell Browser approvals do not create a generic `bash` tool override
- Browser tool and Browser-bearing shell routes can reuse each other's scoped
  grants
- Browser `allow_all_session` still records only Browser scoped grants and
  leaves `allow_all_tools` unset
- Browser approval requests emit identical structured Browser payload fields
  across Browser tool and Browser-bearing shell routes
- Browser prompts may expose Browser context fields for display without
  exposing a frontend-selectable Browser scope list
