# soul.md activation: identity layer in the runtime system prompt

## Summary

`soul.md` is now an activated context file: a small, first-person **identity**
layer (who the agent is, its values, voice, and boundaries), distinct from the
**operating** memory files (`AGENTS.md` / `CLAUDE.md`). It is loaded into the
runtime system prompt on every turn, universal across providers.

## Design

- **Loader** (`crates/puffer-core/runtime/system_prompt.rs`):
  - `load_context_blocks(cwd, filename)` (extracted, shared) reads a filename
    from the project dir plus `~/.claude` and `~/.puffer`, returning one
    `Contents of <path> (<description>):` block per non-empty file, project
    first.
  - `load_memory_prompt_for_filename` wraps those blocks with
    `MEMORY_INSTRUCTION_PROMPT` (unchanged behavior).
  - `load_soul_prompt(cwd)` wraps `soul.md` blocks with the new
    `SOUL_INSTRUCTION_PROMPT` ("The following describes your identity ... this
    is context, not a hard rule.").
- **Injection** (`render_runtime_system_prompt`): after the base template
  renders, the prompt is assembled as `[base template] + [soul identity] +
  [project memory]` - **identity first, then operating instructions**. soul.md
  is appended for all providers; `AGENTS.md`/`CLAUDE.md` keep their existing
  provider convention (AGENTS.md preferred for `openai`, CLAUDE.md otherwise).
- The soul/memory append happens in Rust **after** the prompt render, so it does
  not touch `system-base.yaml` or its `SYSTEM_PROMPT_TEMPLATE` mirror.

## Contracts

- Discovery locations (in order): `<cwd>/soul.md`, `~/.claude/soul.md`,
  `~/.puffer/soul.md`. Empty/missing files are skipped; absent everywhere ->
  nothing injected (no behavior change for repos without a soul.md).
- soul.md is **universal** (provider-independent): identity is not a
  provider-specific convention the way the operating doc is.
- Ordering guarantee: identity precedes project memory in the assembled prompt.

## Compatibility / faithfulness notes

- Versus shipped Claude Code (reverse-engineered): CC delivers memory as a
  **user message after the system prompt** and reads it once at startup
  (re-injecting root `CLAUDE.md` from disk at compaction); puffer instead
  appends memory **into the system prompt** and **rebuilds it every turn**
  (re-reading from disk each turn). puffer's per-turn rebuild makes memory
  survive compaction for free (it lives in the system field the compactor never
  touches) and makes edits take effect on the next turn. This is a deliberate,
  pre-existing puffer design choice, not changed here.
- puffer has no lazy/pointer tier (CC lazy-loads nested `CLAUDE.md`, caps
  `MEMORY.md` to 200 lines / 25KB, and treats topic files as on-demand). Not
  needed for the small `soul.md` / `AGENTS.md` files; a capped/lazy memory tier
  is a possible future follow-up.

## Tests

- `load_soul_prompt_loads_identity_block`: absent -> None; present -> wrapped in
  the identity instruction with a `Contents of ... soul.md` block.
- `render_runtime_system_prompt_injects_soul_before_memory`: soul marker appears
  before the AGENTS.md memory marker in the rendered prompt.
