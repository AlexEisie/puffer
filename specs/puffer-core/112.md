# Planned workflow trigger inference

`WorkflowCreate` and `/workflows append` now accept planned workflow bindings
without requiring the caller to pass a connector slug when the connection name is
the connector's suggested connection slug.

## Design

- Existing connection slugs still resolve through the connection store and
  validate that the connection's connector can produce workflow trigger events.
- Missing connection slugs first honor an explicit `connector_slug`/`--connector`
  value.
- When no connector is supplied, the resolver scans trigger-capable connector
  templates and accepts exactly one template whose suggested connection slug
  equals the requested connection slug.
- Ambiguous or unknown planned connections fail with an error that asks the
  caller to provide the connector slug or run `/connect` first.

## Compatibility

Existing connected workflows are unchanged. Planned workflow bindings continue
to store `connector_slug` so later `/connect` and consumer refresh logic can
associate the binding with the connector after authentication.
