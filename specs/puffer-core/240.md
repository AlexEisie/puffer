# OpenAI Responses Stream Restart

## Summary

OpenAI Responses streaming now restarts the whole sampling request when the SSE
body ends before a terminal `response.completed`-style event.

## Architecture

- `OpenAIResponsesTurnSession` wraps request send, auth refresh, include fallback,
  and SSE parsing in a stream-attempt retry loop.
- `PUFFER_OPENAI_STREAM_MAX_ATTEMPTS` controls full stream restarts; the default
  is 3 and the value is clamped to 1 through 10.
- `PUFFER_OPENAI_STREAM_RETRY_DELAY_MS` controls restart delay and falls back to
  `PUFFER_OPENAI_HTTP_RETRY_DELAY_MS`; the default is 1000 ms.
- Connection-level HTTP retries remain inside each stream attempt.

## Runtime Contract

- `TurnStreamEvent::RetryAttempt` now means live consumers must discard
  in-flight assistant text, thinking, and tool previews from the failed attempt.
- TUI, daemon, benchmark trace, and desktop live state reset their transient
  attempt state before rendering the next attempt.
- TUI session logs stay append-only: any failed-attempt rows already persisted
  are reverted by appending a transcript rewrite.

## Observability

- Each stream restart writes a `puffer::runtime::openai` warning with attempt,
  maximum attempts, retry delay, and the wrapped stream error.
- `PUFFER_OPENAI_SSE_TRACE_PATH` can capture raw SSE reads and parser decisions
  such as EOF before terminal event for endpoint-level diagnosis.

## Compatibility

- Successful responses keep the existing final `AssistantTurn` semantics.
- Reasoning include validation fallback still runs inside a stream attempt.
- OpenAI provider-stream tool previews, such as native `web_search_call`, carry
  metadata so consumers do not treat them as durable local tool completions when
  computing retry baselines.
- Provider-stream previews do not consume pending local tool-call rows; local
  tool completions advance the retry baseline only after they are observed.
