# Retry OpenAI SSE provider failures without parse-error wrapping

## Why

ChatGPT Codex can return HTTP 200 with an SSE `response.failed` event whose
payload carries a provider code such as `server_error`. Puffer previously
converted that event into a plain `anyhow` string and then wrapped it as
`failed to parse SSE response`, which made an upstream transient look like a
local parser bug and prevented the OpenAI transport retry helper from matching a
structured retryable failure.

## Change

- `openai_sse` now emits a structured `OpenAISseApiError` for
  `response.failed` events.
- Retry classification treats `rate_limit_exceeded`, `server_error`, and
  `overloaded` SSE failures as retryable across the HTTP SSE and WebSocket
  transports.
- SSE parse wrappers preserve provider-side failed events as provider errors
  and only add `failed to parse SSE response` context for real malformed or
  incomplete SSE bodies.

## Result

Transient Codex backend `server_error` events use the existing OpenAI HTTP retry
budget. If all attempts fail, the user sees the provider failure itself instead
of a misleading SSE parser error.

- `parse_openai_sse_response_classifies_server_errors_as_retryable`
- `parse_openai_sse_response_surfaces_failed_events`

- `is_retryable_ws_error_matches_sse_failed_events`
