# Connected Image Provider Capability Expansion

## Summary

Image generation capability resolution should list every provider that is both
connected and executable for image generation. Providers that only support chat,
image input, or unverifiable router defaults must stay hidden.

This update deliberately does not preserve the fixed OpenAI-shaped media
settings schema. The long-term contract is capability-driven: desktop, daemon,
Tauri preview, and the agent image-generation tool all consume the same exact
capability records and execute through the same adapter registry.

The reviewed scope is intentionally smaller than a generic media platform. The
first stable target is connected cloud providers with documented image
generation APIs, plus router/gateway image-output discovery where the provider
returns structured model metadata. Local experimental image generation is an
extension point, not a default background probe.

## Capability Rule

`list_media_capabilities(kind=image)` returns a provider only when all of these
conditions hold:

1. The provider is connected.
2. The provider has image-generation capability metadata from static resources,
   trusted provider discovery, or a successful local probe.
3. At least one concrete image model is known.
4. The model is executable by an implemented adapter.
5. The model id is not `auto`, wildcard-like, or inferred from a chat route.

Connected means:

- cloud providers have non-empty credentials in the Puffer auth store;
- router and gateway providers have credentials and trusted discovery for
  image-output models;
- local providers have a reachable endpoint and a successful image-generation
  probe;
- auth-free non-local providers require explicit static media descriptors.

## Capability DTO

The daemon should expose parameterized media capabilities instead of OpenAI-only
fields:

```text
provider_id
provider_display_name
model_id
model_display_name
kind = image
operation = generate
adapter
status = available | unavailable
parameters
defaults
source = static | provider_discovery | local_probe
reason
checked_at_ms
```

`parameters` is a small select-only schema in the initial implementation, not
a free-form request template or generic form system. Each parameter has a name,
allowed string values, a default, and an optional adapter request-field name.
Add `number`, `boolean`, or free-form string controls only when a connected
provider needs them for a tested user-visible capability.

Saved image settings should store:

```text
provider_id
model_id
adapter
parameters
```

Generation must re-resolve the saved capability and validate every saved
parameter before sending an HTTP request.

## Adapter Families

Implement small API-shape adapters rather than provider-specific desktop logic
or a generic request-template engine.

- `images_json` executes synchronous JSON image endpoints that return
  OpenAI-like `data[].b64_json` or `data[].url` images. It covers OpenAI,
  BytePlus ModelArk, Zhipu, xAI, Vercel image-only models, and vLLM-Omni only
  when an explicit local capability later proves available.
- `chat_image_output` executes router or gateway chat models with documented
  image output metadata. Target providers include OpenRouter and Vercel
  multimodal/image-output chat models.
- `minimax_image` executes MiniMax and MiniMax CN `/v1/image_generation`.

Do not add an adapter enum value before the adapter is executable and tested.
Do not add arbitrary request templates. `images_json` may support only a small
allowlist of request fields: `model`, `prompt`, `n`, `size`, `quality`,
`output_format`, `response_format`, `aspect_ratio`, and `resolution`.

## Provider Coverage

The current provider list should resolve as follows:

| Provider | Target treatment |
| --- | --- |
| `openai` | Show when connected; static exact image models use `images_json`. |
| `byteplus` / `bytepluses` | Show when connected and exact Seedream descriptors are present; use `images_json`. |
| `zhipu` | Show when connected and exact GLM/CogView descriptors are present; use `images_json`. |
| `xai` | Show when connected and Grok Imagine descriptors are present; use `images_json`. |
| `vercel-ai-gateway` | Show when connected and trusted discovery or static descriptors identify image-output models; use `images_json` for image-only models and `chat_image_output` for chat image output. |
| `openrouter` | Show when connected and provider discovery returns concrete image-output models; use `chat_image_output`. |
| `minimax`, `minimax-cn` | Show when connected and exact `image-01` descriptors are present; use `minimax_image`. |
| `vllm` | Hide by default. Show only in a later local-provider slice when an explicit local image descriptor and manual probe confirm vLLM-Omni behavior. |
| `ollama` | Hide by default. Experimental image generation can be added later with a dedicated local-provider slice. |
| `worldrouter` | Hide until official image-generation API or trusted image-output metadata exists. |
| `anthropic`, `groq`, `cerebras`, `kimi-coding`, `kimi-openai`, `llama-cpp`, `lmstudio`, `minicpm5` | Hide because current evidence is chat-only, image-input-only, text-only, or otherwise not executable image generation. |

The `bytepluses` provider id should be normalized to `byteplus` as part of this
breaking change, with auth and resources using the same canonical id.

## Performance

Settings snapshot loading must not perform media discovery. Capability
resolution is lazy and runs when image settings are opened or when generation
validates a saved selection.

Static capabilities are returned without network calls. Provider discovery uses
bounded concurrency, short timeouts, and TTL caches:

- cloud discovery cache: minutes, because model lists and capabilities drift but
  not per render;
- per-provider failure isolation: one provider failure cannot hide other
  available image providers.

Local image-generation probes are not part of default settings loading or the
initial connected cloud-provider implementation.

## Stability

The runtime must never infer image capability from model names, provider names,
chat API family, or input modalities. It must not expose router `auto` models.

Adapters must download returned image URLs into local artifacts before reporting
success. Temporary provider URLs are metadata only. Secret-bearing headers,
query parameters, and auth tokens must be redacted from all returned errors.

## Tests

Core coverage should include:

- each adapter succeeds against mock HTTP fixtures;
- each adapter handles provider errors and secret redaction;
- URL and base64 outputs both persist local artifacts;
- stale provider/model/adapter selections fail before HTTP;
- connected providers without image descriptors stay hidden;
- image-input-only models stay hidden;
- router `auto` models stay hidden;
- discovery timeout for one provider does not suppress static providers.
- the initial parameter schema stays select-only.
