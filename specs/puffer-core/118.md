# 104. Computer-Use via CUA MCP Bridge

## Summary

Puffer gains **computer-use** — driving a graphical desktop by screenshot +
mouse/keyboard/window actions — by bridging to [CUA](https://github.com/trycua/cua)'s
`computer-server` (MIT) over MCP. A containerised XFCE Linux desktop runs the
server, which exposes ~44 `computer_*` actions on an MCP streamable-HTTP endpoint
at `/mcp`. Puffer connects to it as an ordinary external MCP server; **Puffer's own
model loop is the agent** that decides when to screenshot, click, and type. CUA's
own agent loop is not used.

Two phases are documented below: **Phase 1 (bridge)** uses CUA's Python
`computer-server`; **Phase 2 (native)** replaces it with CUA's Rust
`cua-driver-rs` (no Python) for headless container deployment.

## Design

### Topology

```
puffer (vision model loop = the agent)
   │  MCP streamable-HTTP  (rmcp client, runner_mcp/transport.rs)
   ▼
CUA computer-server  (Docker xfce sandbox, port 8000, /mcp)
   ▼
XFCE Linux desktop   (1024x768; noVNC on :6901 to watch)
```

`scripts/cua-computer-sandbox.sh up` brings up the sandbox (image
`trycua/cua-xfce:latest`), waits for `GET /status`, and prints the registration
command. `down`/`status`/`shot`/`register` manage and inspect it.

### Wiring

1. **Sandbox**: `scripts/cua-computer-sandbox.sh up` — XFCE desktop +
   computer-server. Health: `GET /status` → `{status:ok, features:[agent,mcp]}`.
2. **MCP registration** (per workspace/user; not bundled because the endpoint is
   only live while the sandbox runs):
   `puffer mcp add cua-computer http://localhost:8000/mcp/ -t http -s user`.
   Tools surface as `mcp__cua-computer__computer_*` and dispatch like native tools.
3. **Driving model**: any vision-capable provider configured in
   `resources/providers/`. The model must accept image tool-results. Verified with
   an OpenAI-Responses-API gateway (`api: openai-responses`, custom `base_url`).

### Load-bearing contract: image tool-results

`computer_screenshot` (and other capture actions) return an MCP result whose
content carries an image part. Puffer surfaces this to the runtime as
`non_text_content: [{ "type": "image", "data": "<base64>" }]` and forwards it to
the model as an image input. **This is the integration's critical seam** — without
image forwarding the model is blind and the loop is useless. A single
`puffer non-interactive` turn performs multiple internal tool-use rounds
(screenshot → reason → act → screenshot …), bounded by `--max-tool-calls`.

### Known quirks (carried into Phase 2)

- **Streaming-only gateway output**: some OpenAI-Responses gateways only emit
  message content via streaming events (`response.output_text.delta`); the
  non-streamed `output[]` is empty even on `status:completed`. Puffer consumes the
  Responses API as a stream, so this is transparent — but a non-streaming probe of
  such a gateway will look like an empty response.
- **`computer_press_key "Return"`**: CUA's key-name mapping rejects `Return`
  ("Unknown key"); models work around it by typing a newline via `computer_type`.
  The native port should normalise key names.

## Phase 2: native (`cua-driver-rs`)

CUA ships a Rust port, `cua-driver-rs`, with a Linux backend (pure-Rust
`x11rb`/`image`/`tiny-skia`; AT-SPI accessibility) and its own stdio MCP server
(`cua-driver mcp`, 32 tools: `screenshot`/`click`/`type_text`/`launch_app`/
`list_windows`/`get_accessibility_tree`/…). The native path drops the Python
`computer-server` sidecar entirely — suited to headless AgentEnv Linux containers.

### Topology (native)

```
puffer ──(stdio: docker exec -i)──▶ cua-driver mcp (Linux container) ──▶ Xvfb desktop
```

`docker/cua-driver-native/` builds a ~480MB image (linux/arm64 builds natively on
Apple Silicon): stage 1 compiles `cua-driver`, stage 2 is a Debian + Xvfb +
openbox desktop. The MCP server is reached over stdio by pointing Puffer's stdio
transport at `docker exec`:

```yaml
# ~/.puffer/resources/mcp_servers/cua-native.yaml
id: cua-native
transport: stdio
target: docker exec -e DISPLAY=:99 -i cua-native-sandbox cua-driver mcp --no-overlay --no-daemon-relaunch
```

`--no-daemon-relaunch` forces in-process (no daemon needed on Linux).
`--no-overlay` is **required** (see quirks). Tools surface as `mcp__cua-native__*`.

### Verified

puffer (vision model) called `screenshot` / `get_accessibility_tree` /
`list_windows` / `click` / `type_text` against the container and read screenshots
back correctly — fully native, no Python.

### Native quirks (cua-driver Linux backend)

- **Agent-cursor overlay renders opaque black** under Xvfb (RGBA override-redirect
  window with no compositor to alpha-blend it). It covers the screen, so
  `screenshot` returns all black. Launch `cua-driver mcp` with **`--no-overlay`**
  (or run a compositor like `picom`).
- **`type_text` uses XSendEvent, not XTest**, with two consequences on Linux:
  (1) target apps that ignore synthetic events drop the input — xterm needs
  `*allowSendEvents:true` (baked into the image's xterm launch); (2) modifier
  state isn't applied, so SHIFTed characters are wrong (`NATIVE_CUA_WORKS` arrives
  as `native-cua-works`) and `\n` doesn't submit. A robust fix upstream would route
  Linux key synthesis through XTest. Until then, prefer `launch_app` + clicks +
  lowercase/no-symbol text, or drive apps that honor SendEvents.

## Phase 3: desktop (vendored, drives the user's own machine)

For the **Puffer desktop app**, computer-use drives the *user's own computer*
(not a sandbox). CUA's `cua-driver` is **vendored** into the repo at
`vendor/cua-driver/` (MIT — `vendor/cua-driver/LICENSE` + `PUFFER_NOTICE.md`;
its own cargo workspace, excluded from Puffer's build). On macOS this is CUA's
flagship **background driving** — it acts on a target window via per-window APIs
without stealing the cursor/focus/Space.

### Wiring (zero manual steps for the user)

`scripts/build-cua-driver.sh` builds the vendored driver, installs it to
`$PUFFER_HOME/bin/cua-driver`, and writes a user-scoped MCP manifest:

```yaml
# $PUFFER_HOME/resources/mcp_servers/cua-computer.yaml  (generated, not committed)
id: cua-computer
transport: stdio
target: <abs>/bin/cua-driver mcp --no-daemon-relaunch
```

The manifest is **generated** (not committed) because stdio `target` is not
env-expanded and needs an absolute path. The desktop daemon
(`apps/puffer-desktop` → `puffer daemon`) auto-loads it, so the model gets
`mcp__cua-computer__*` tools that drive the real desktop. Productization TODO:
have the Tauri app run the build/install step on first launch (or ship
`cua-driver` as a Tauri `externalBin` sidecar) so end users skip even the script.

### macOS permissions

`cua-driver` needs **Accessibility** + **Screen Recording** (System Settings ›
Privacy & Security). Probe non-invasively with the `check_permissions` tool.
For a signed/shipped app these are requested on first use; CUA itself ships a
signed `CuaDriver.app` for clean TCC attribution.

## Choosing a path

- **Bridge (Phase 1)**: richest/most-mature actions, but a ~4GB Python image.
- **Native container (Phase 2)**: ~480MB, no Python — best for headless
  container deployment, with the input caveats above.
- **Desktop vendored (Phase 3)**: drives the user's own machine; macOS background
  driving; vendored MIT source, auto-registered. This is the desktop-user path.

## Testing

- `scripts/cua-computer-sandbox.sh up && ./scripts/cua-computer-sandbox.sh shot`
  — sandbox + capture work standalone.
- `puffer mcp list` shows `cua-computer [http]`.
- E2E: `puffer non-interactive --provider <p> --model <m> --user-message
  "Open a terminal and run echo HELLO" --max-tool-calls 14` — model launches the
  app, types, executes; verify via `shot` or noVNC.
