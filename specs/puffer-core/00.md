# puffer-core

## Purpose

`puffer-core` is the shared application engine for Puffer Code. It owns command
registration and dispatch, app state, provider-turn orchestration, transcript
conversion, tool execution, permission handling, compaction, resource-backed
prompts, and runtime service integration.

## Design

The crate is organized around these boundaries:

- `AppState` is the mutable session state used by CLI, TUI, daemon, connectors,
  and tests.
- `CommandSpec` and command dispatch define slash-command behavior.
- Provider adapters convert selected provider/model descriptors into a
  provider-neutral `TurnSession`.
- The agent loop drives one provider turn, tool execution, compaction, usage,
  reflection, and final transcript updates.
- Tool execution is mediated by a registry, request-scoped filters,
  permissions, hooks, and runtime handlers.
- Resource-backed prompts and skills keep system behavior editable without
  hard-coding large prompt bodies.

## Logic

State restoration must replay session transcript events and state snapshots
without replacing the append-only session log. State mutation should emit
snapshot events when changes affect future behavior.

Command dispatch must:

1. Resolve command aliases and resource-provided skills.
2. Decide whether a command is prompt-backed, local, or UI-only.
3. Record `CommandInvoked`.
4. Render prompt resources when needed.
5. Execute provider turns for prompt commands.
6. Apply local state/config/session changes for local commands.
7. Leave UI-only commands for the frontend surface.

Provider turn execution must:

- Resolve the current provider and model.
- Select the adapter for Anthropic Messages, OpenAI Responses, Azure OpenAI
  Responses, OpenAI Codex Responses, or OpenAI Chat Completions.
- Convert the transcript into neutral conversation items.
- Inject resource-backed system prompt and memory context.
- Compact before or after turns when limits require it.
- Stream thinking, text, tool-call, usage, retry, permission, and finish events.
- Execute tool calls through the permission and hook pipeline.

## Contract

`AppState` is the in-process state contract. It must preserve current
provider/model, theme, prompt color, effort, fast mode, plan mode, sandbox,
working directories, remote settings, team/statusline/Vim state, read-state,
permissions, tasks, and transcript.

The supported slash-command surface is the contract exposed by
`supported_commands` plus resource-backed `/skill:<name>` entries. Aliases must
remain in `puffer-core` so all frontends agree.

Tool permission decisions must support allow, deny, and ask outcomes, with
session allow-all behavior applied only after an explicit permission flow.

Anthropic-specific compatibility details must remain in the Anthropic transport
and adapter path. They must not be erased by over-generalizing provider code.

The crate must not add telemetry, feedback reporting, or analytics side effects.

## Architecture Details

`puffer-core` is the application boundary below all frontends. Frontends may be
CLI, TUI, daemon, desktop API, or connectors, but they all rely on the same
state and command/runtime behavior here.

Internal architecture is layered:

- State layer: `AppState`, transcript reconstruction, config/session snapshots,
  and task/background state.
- Command layer: slash command catalog, aliases, prompt command preparation,
  local command effects, and UI command declarations.
- Prompt/resource layer: system prompt assembly, command prompts, skills,
  memory files, and resource-driven tool metadata.
- Runtime layer: provider adapter selection, transcript conversion, compaction,
  streaming events, retry behavior, and usage accounting.
- Tool layer: request-scoped filtering, permission checks, hook execution,
  runtime tools, provider tools, and `puffer-tools` dispatch.

No frontend should bypass these layers for behavior that affects transcript,
permissions, provider selection, or command semantics.

## Contract Details

- Command contract: `supported_commands` and aliases are canonical; generated UI
  surfaces must use this registry instead of hard-coded duplicate lists.
- State contract: `AppState` must be restorable from session-store events plus
  snapshots and must not require opaque private state to resume.
- Runtime contract: turn execution emits structured stream events for thinking,
  text, tool requests, tool results, permissions, retries, usage, reflection,
  and finish status.
- Provider contract: adapters implement provider-specific transport while the
  loop remains provider-neutral.
- Tool contract: visibility filtering, approval, hooks, execution, and result
  shaping must be applied consistently across Anthropic and OpenAI paths.
- System prompt contract: base prompt, tool descriptions, memory, environment,
  and session guidance are assembled from resources and state with explicit
  fallbacks.
- Failure contract: provider, tool, permission, compaction, and command errors
  are returned as structured outcomes or transcript-visible errors; they should
  not corrupt session replay.
- Exclusions: no telemetry/reporting side effects and no provider-specific
  simplification that breaks Anthropic compatibility.
