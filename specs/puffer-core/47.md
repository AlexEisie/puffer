# Telegram multi-account connection slugs

- The `Telegram` internal tool accepts `connection_slug`/`connection` and
  `account_slug`/`account` on every action. Omitted slugs preserve the existing
  `telegram-user` default.
- Passing the connector slug `telegram-login` as a connection alias also maps
  to `telegram-user` for compatibility with older direct calls.
- Each non-default Telegram connection slug starts a synthetic subscriber
  manifest derived from the bundled `telegram-user` manifest with `id` and
  `topic` set to the connection slug.
- The connection slug is the stable Telegram account identity. Import-time
  `account_index` only selects a slot from local Telegram storage.
- Non-default Telegram state is isolated under
  `<user_config_dir>/telegram-accounts/<connection_slug>/`, so imported
  Desktop accounts, QR login sessions, and phone-code login sessions do not
  overwrite each other.
- Successful Telegram login/import creates the matching authenticated
  `telegram-login` connection record automatically, returning the connection
  and whether it was newly registered.
- `WorkflowCreate` starts the selected Telegram subscriber for enabled
  `telegram-login` connections before persisting the workflow, so new
  workflows can receive events immediately.
- Direct `ConnectorAct` now injects the resolved connection slug into the
  action input and synthetic trigger envelope before using built-in fallback
  dispatch, preserving account selection for subscriber-backed actions.
