# Surface top-level OpenAI SSE `error` events instead of masking them

## Why

Some OpenAI-compatible endpoints return HTTP 200 with a top-level SSE `error`
event — e.g. `{"type":"error","code":"context_too_large","message":"Your input
exceeds the context window of this model."}` — and then close the connection
without a terminal `response.*` event. `specs/puffer-core/238.md` handled
`response.failed`, but the parser had no arm for this standalone `error` event,
so it reached EOF with `terminal == false` and bailed with `stream closed before
response.completed`. That string was then wrapped as `failed to parse SSE
response` and classified as a retryable transport failure — so a deterministic
error (e.g. a context overflow) was retried up to 6× with the same input, and
the user never saw the real `code`/`message`.

## Change

- `openai_sse` now emits a structured `OpenAISseApiError` for top-level `error`
  events via `OpenAISseApiError::from_error_event`, reading `code`/`message`
  from the document root (and accepting proxies that nest them under `error`).
- Because it is a typed provider error it short-circuits the `failed to parse
  SSE response` wrap and flows through the existing `is_retryable()` whitelist:
  `context_too_large` is non-retryable, while transient codes (`server_error`,
  …) delivered as a top-level `error` event remain retryable.
- The surfaced message contains "exceeds the context window", so
  `overflow::is_context_overflow` can recognize it downstream.

## Result

A top-level overflow error surfaces as `context_too_large: Your input exceeds
the context window of this model` immediately, with no wasteful retries, instead
of a misleading `stream closed before response.completed`.
