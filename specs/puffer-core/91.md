# Resource Watcher Debounce Replay

## Design

The resource watcher still debounces bursty file events, but it now checks
whether the reload signal is already latched before suppressing an event inside
the debounce window.

## Logic

- If a signal is already pending, additional events inside the 200 ms debounce
  window are coalesced as before.
- If the TUI has consumed the signal, a follow-up event inside the same window
  can raise it again.
- This prevents a directory-create event from causing a reload before a new
  `SKILL.md` is written and then suppressing the actual file-write event.

## Compatibility

Existing watcher roots and path filtering are unchanged. The change only makes
hot reload more reliable when editors or tests create resource directories and
files in quick succession.
