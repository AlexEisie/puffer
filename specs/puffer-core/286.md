# Reactive context-overflow recovery in the agent loop

## Why

The agent loop only had proactive, threshold-gated compaction, derived from the
model catalog's `context_window`. When an OpenAI-compatible endpoint's real
window is smaller than the catalog value (proxy / gateway / Azure / self-host),
the endpoint rejects the request far below puffer's threshold, so compaction
never fires and the turn just errors out. There was no reactive "compact on
overflow" path in the general session — only the pentest harness had one. (Even
calling `compact_conversation_with` from the error branch would no-op, because
it re-checks the same catalog threshold.)

## Change

- `compact_conversation_with` is split into a shared `compact_conversation_inner`
  with a `force` flag; new `force_compact_conversation_with` bypasses the two
  context-window threshold early-returns and targets an empty transcript in the
  Phase 3 drop loop. Phase 2 (AI summary) stays the primary reducer.
- On a turn error, the agent loop runs `overflow::is_context_overflow` on the
  error and, on a match, force-compacts the transcript and retries the turn.
  Bounded by `MAX_OVERFLOW_RECOVERIES` *consecutive* attempts — the counter
  resets after any successful turn, so a long session can recover as many times
  as it needs over its lifetime — and gated on the compaction actually shrinking
  the transcript so an already-minimal turn can't loop.

Depends on `specs/puffer-core/285.md`: the real `code`/`message` must surface
first, otherwise the masked "stream closed…" string is never recognized as an
overflow.

## Result

A session behind a smaller-than-catalog endpoint recovers automatically: the
request grows to the wall, overflows once, force-compacts (a sharp request-size
drop), retries, and continues — repeatedly across a long session without
exhausting the budget.
