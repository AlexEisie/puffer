# Contact Infer Proposal UX

## Summary

The Contacts screen presents inferred contacts as proposal cards until the user
explicitly chooses one and saves it.

## Design

- `inferContacts` returns a partial contacts snapshot without `savedCount`.
- The Infer dialog always reports proposal counts and renders proposal cards
  when proposals are available.
- Running inference does not select a new saved contact, because no saved
  contact exists until the user clicks `Use` and confirms the create dialog.
- The fake daemon mirrors the daemon contract by persisting proposals,
  deduplicating inferred identities, and filtering proposals that overlap saved
  contacts.

## Compatibility

Existing saved contacts remain visible when inference refreshes the snapshot.
Persisted proposal rows continue to hydrate on screen reload and modal reopen.
