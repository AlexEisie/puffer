# Workflow Fuzz Surface Terminology

The desktop fuzz coverage model now treats Workflows as the canonical GUI
surface name.

- Fuzz route, tree, seed, shard, and replay action ids use `workflows` and
  `open-workflows` instead of the removed legacy route naming.
- Replay generation opens only the Workflows navigation item and delays the
  real `workflow_list` daemon method when simulating workflow refresh races.
- Settings connector coverage is represented in the fuzz route tree so the
  dynamic AskUserQuestion setup page remains part of desktop UI/UX risk
  modeling.
- The workflow provider switcher's DOM hook now uses `data-workflow-provider`;
  the serialized daemon workflow contract still uses `workflow.pipeline.*`.
