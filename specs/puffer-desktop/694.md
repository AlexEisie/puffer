# Task Settings Modal

## Scope

The desktop Tasks screen task configuration dialog now follows the Paper task
settings designs for Monitor, Contacts, and Rules and memory. The modal must
remain frontend-only and keep all existing daemon contracts unchanged.

## Change

- The dialog title changes to `Task settings` with a 760px by 760px modal shell
  and a left navigation rail for `Monitor`, `Contacts`, and
  `Rules and memory`.
- The Monitor tab keeps the existing monitor create/update/delete flow, but
  presents account and model controls in the compact layout from the design.
- The Monitor tab add action uses the design primary button treatment: a
  48px by 30px black button with 6px radius, independent of the shared button
  defaults.
- The Contacts tab moves contact scope selection into a table with checkbox,
  contact, description, and action columns. Contact selection remains draft
  state until the monitor is updated through the existing monitor API.
- Contacts table checkboxes use the 16px filled square treatment from the
  design, and row edit actions use a dedicated 30px circular icon button
  instead of the generic task settings button size.
- The Rules and memory tab shows monitor-owned filter rules and the existing
  monitor memory editor in one surface. Memory save still uses the existing
  `task_monitor_memory_save` contract.
- Rules and memory primary actions use the same black 30px-high design button:
  the ignore-rule `Add` action stays disabled until backend support exists,
  and `Save memory` is clickable even when the draft matches the current file
  content so users can explicitly trigger a save.
- The modal body uses fixed header sizing, a scrollable content pane, and
  responsive tab behavior so dense contacts or small viewports keep bottom
  content reachable.
- Tasks screen and task settings text styles do not use font sizes below 12px,
  keeping labels, metadata, chips, history rows, and modal fields readable.

## Backend Gap

Manual structured ignore-rule creation and deletion are represented as disabled
frontend controls because the daemon does not currently expose a structured
rule create/delete RPC. Adding that behavior requires a backend API contract
before the controls can persist changes.

## Compatibility

No Rust or daemon logic changes are required. Existing task monitor creation,
contact scope selection, monitor deletion, filter-rule display, and monitor
memory editing behavior are preserved.
