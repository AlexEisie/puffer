# Canvas State Submit Button

## Summary

Inline Canvas controls now include a reusable `Submit choices` action in the
selection area. After users finish changing controls, the button sends a normal
chat follow-up that asks the agent to read the saved state with `CanvasState`
and continue, so users no longer need to type a second prompt such as "read the
state now."

## Behavior

- `InlineCanvas` still saves changed control values through
  `updateCanvasState`.
- `CanvasSubmitButton` owns the reusable button and state badge presentation.
- The submit action is rendered in the canvas body below the selectable
  controls, not in the header metadata row.
- The submit row only shows active save/submit states, avoiding persistent
  warning badges while the user is still changing controls.
- If a user edits before the Canvas id has hydrated, Desktop keeps the patch
  pending and saves it once the id is available instead of surfacing an error.
- The submit button is disabled as `Preparing...` until the Canvas id and
  session id are available.
- If Desktop rejects a follow-up because the session is not ready for a new
  turn, the submit row returns to idle instead of showing a red error badge.
- Canvas save failures now show the daemon error text beside the submit action
  so users can distinguish stale daemons, missing state files, and session
  mismatches.
- Canvas tool bodies use a taller viewport-aware height cap than generic tool
  output so common selection screens fit without internal scrolling.
- Multiple rapid control changes are merged into one state patch before saving.
- Clicking `Submit choices` flushes any pending state patch before submitting
  the follow-up.
- The submitted message includes the Canvas id and tells the agent to call
  `CanvasState`, briefly confirm the selected values, and continue without
  asking the user to repeat the choices.
- Repeated button clicks with an unchanged Canvas value do not submit duplicate
  follow-ups.

## Compatibility

CanvasState storage and tool schema are unchanged. The button is limited to the
inline Desktop renderer because non-Desktop hosts do not pass a canvas submit
callback into the component.
