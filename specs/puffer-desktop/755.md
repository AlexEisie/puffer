# Monitor Rule Condition Editor

## Design

The Tasks settings Rules tab labels the monitor condition section as `Filter
rules` and uses a condition editor instead of free-form keyword chip inputs.
Each monitor binding renders its own include and exclude conditions:

- `Only create tasks when` adds include rules.
- `Skip tasks when` adds exclude rules.
- `Message text` submits keyword rules against `Event.text`.
- Connector-specific fields render as plain `Message detail` options directly
  after `Message text`, without a separate option-group label or indentation.
- Field rules submit connector-specific paths from `monitor_rule_schema.fields`.

Saved regex and jq rules are displayed as readable condition chips, using
schema labels and value labels when available.

## Behavior

- Switching the selected monitor switches the displayed rules, schema fields,
  and associated memory together.
- Changing `Message detail` resets the condition and value controls to valid
  defaults for that field.
- Changing `Condition` refreshes the value control so stale input is not
  reused under a different operator.
- Without a schema, the editor still exposes `Message text` keyword rules and
  hides payload field choices.
- Message text requests include the selected operator, defaulting to `contains`
  in the API helper for compatibility.
- Payload field rules continue to send `field`, `operator`, and `value` from
  connector schema metadata.

## Layout

Monitor rule conditions render as sorted, content-sized pills in a wrapping
chip list instead of full-width rows. Conditions are sorted by displayed title
with locale-aware, case-insensitive comparison. Each pill sizes to its text plus
the remove action, truncates safely when needed, and multiple short conditions
share a row before wrapping.

The condition builder uses a three-column control grid and a separate action
row so Cancel and Add condition remain inside the rule group. Narrow screens
collapse the builder to one column.

## Pill Colors

Condition pills use field-aware tones so users can scan mixed payload
conditions quickly. `Message text` keeps a dedicated text tone because it always
maps to `Event.text`. Connector fields receive tones from the selected schema
order, with distinct tones for the first 20 fields before the palette repeats.
Unknown fields fall back to the first payload tone.

## Compatibility

This is a Desktop UI and request-shaping change. Workflow snapshots,
subscription storage, and daemon skip-rule persistence continue to use the
existing `ignore_filters` field. Monitor rule add/delete request modes remain
`include` and `exclude`.
