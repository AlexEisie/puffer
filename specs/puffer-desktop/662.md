# HTML5 Chat File Drop

## Scope

Desktop chat should use the browser HTML5 drag/drop file model as the only
attachment drop path.

This replaces the native Tauri path-drop bridge. The chat composer needs file
objects for preview and future upload staging, not native filesystem paths.

## Decision

Set the main Tauri webview to `dragDropEnabled: false` and keep the existing
DOM `DataTransfer.files` handlers as the single drag/drop implementation.

The optimized flow is:

```text
File picker
DOM file drop
  -> FileList | File[]
  -> addAttachmentFiles()
  -> ComposerAttachmentDraft[]
  -> AttachmentPreviewStrip
  -> submitted message metadata
```

The removed flow is:

```text
Tauri onDragDropEvent(paths)
  -> custom window event
  -> daemon read_file
  -> base64/utf8 decode
  -> new File()
```

## Rationale

Tauri native drop events expose local paths and are useful for file-manager
style workflows. Chat attachment UX is different: it only needs the files the
user intentionally dropped into the composer surface. HTML5 drag/drop already
provides those files without any daemon read, path allowlist, MIME re-inference,
or base64 conversion.

This keeps the implementation smaller and removes a security/performance
detour:

- no custom Tauri event bridge;
- no frontend path-to-file conversion;
- no daemon reads for dragged files;
- no failure mode for dropped files outside daemon allowed roots;
- one code path for file picker and drag/drop validation.

## UX Contract

- Dragging files over the active chat surface shows the existing centered drop
  overlay.
- Dropping files adds image thumbnails and file cards to the composer draft
  strip.
- The same limits apply to picker and drop attachments: 10 files maximum and
  20 MiB per file.
- Non-file drag payloads do not activate the overlay.
- Drops are ignored while the composer is disabled.
- Submitted user messages continue rendering attachment cards instead of raw
  `[Image: sample.png]` or `[File: report.pdf]` placeholders.

## Architecture

`ConversationView.svelte` remains the owner of chat drag state:

- DOM `dragenter`, `dragover`, `dragleave`, and `drop` handlers live on the
  `.pf-chat` surface.
- The handlers only accept payloads where `DataTransfer.types` includes
  `Files`.
- `filesFromDataTransfer()` returns the `File[]` that flows into
  `addAttachmentFiles()`.

`attachments.ts` remains focused on browser file objects:

- file input accept list;
- attachment limits;
- `FileList | File[]` validation;
- object URL previews for images;
- cleanup helpers.

Tauri configuration owns the desktop WebView behavior:

- main window sets `dragDropEnabled: false`;
- this disables Tauri's native file-drop handler so HTML5 file drag/drop can
  reach the frontend.

No attachment staging API is introduced in this change. When providers later
support real binary attachments, staging should consume the existing `File`
objects directly rather than reintroducing native path reads.

## Error Handling

Browser-provided `File` objects are validated before draft creation:

- files larger than 20 MiB are rejected with the existing size message;
- files beyond the 10-attachment limit are rejected with the existing count
  message;
- empty drops produce no state change.

There is intentionally no separate unreadable-path error because path reads are
not part of this design.

## Testing

Keep and rely on the existing chat UI drop test that dispatches a browser
`DataTransfer` with image and PDF files. It should verify:

- overlay appears on file drag;
- drop adds previews to the composer;
- submitted metadata includes both attachments;
- submitted chat messages render cards, not placeholder text.

Add a small static configuration test that reads
`apps/puffer-desktop/src-tauri/tauri.conf.json` and asserts the main window has
`dragDropEnabled: false`. This protects the one Tauri-specific requirement
without adding another UI path.

## Non-Goals

- Directory drops.
- Native path access in the frontend.
- Tauri `fs` plugin permissions for arbitrary dropped paths.
- Attachment staging or binary provider transport.
- Separate drop zones inside the transcript.
