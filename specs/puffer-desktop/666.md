# puffer-desktop Update 666: Persistent Chat Attachments

## Summary

Desktop chat attachments are staged into the local session store before submit,
then reloaded from structured timeline attachment DTOs after refresh.

## Behavior

- Composer picker/drop behavior and draft previews remain unchanged.
- Local desktop submit stages draft files with `stage_chat_attachment` before
  starting the turn.
- Remote sessions reject local file attachments with a clear status message.
- `runAgentTurn` sends `attachmentIds`, not full attachment metadata or bytes.
- The optimistic user row uses staged refs merged with transient draft
  `previewUrl` values.
- Persisted pending messages still strip object URLs.
- Refreshed user messages render attachment cards from structured
  `attachments` DTOs.
- Fallback `[Image: name]` and `[File: name]` prompt lines are hidden only when
  structured attachments exist.
- Clicking a refreshed image attachment reads preview bytes on demand through
  the Tauri preview command.
- Missing or unsupported attachments open the existing unavailable detail
  overlay.

## Boundaries

- No production daemon WebSocket file-byte transport.
- No placeholder parser that synthesizes cards from message text.
- No download, system-open, rich PDF/document preview, cloud sync,
  cross-session dedupe, or background attachment GC.
- Fake daemon preview responses and frontend staging hooks are dev/test-only.

## Verification

Playwright coverage should verify submit payload IDs, hard-refresh card
survival, hidden fallback text, available image preview after refresh, and
missing attachment unavailable detail.
