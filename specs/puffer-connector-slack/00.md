# puffer-connector-slack

## Purpose

`puffer-connector-slack` connects Slack Socket Mode events to Puffer Code
sessions. It supports app mentions, direct messages, and threaded replies.

## Design

The crate uses `slack-morphism` for Slack Socket Mode and Web API calls.
Slack-specific event parsing and reply delivery stay in this crate; shared
session and agent behavior stays in `puffer-connector-core`.

Config includes bot token, app token, allowed users, mention requirements,
group-key policy, and optional welcome text.

## Logic

Message handling must:

- Ignore bot-originated messages.
- Enforce `allowed_users` by Slack user ID when configured.
- Treat direct messages as addressable.
- Require app mention in channel contexts by default.
- Strip mention tokens from the message before dispatch.
- Include `thread_ts` in the conversation key when Slack thread context exists.
- Dispatch built-in connector commands before normal agent turns.
- Send responses back in the same Slack thread when applicable.
- Split outbound text with the Slack splitter limit.

## Contract

The connector requires a Slack bot token and Socket Mode app token. Token
validation and workspace app installation are outside this crate.

Thread preservation is part of the user-visible contract: replies to threaded
messages must stay in the originating thread unless Slack omits thread context.

Slack event payloads must be reduced to the shared connector-core input model
before agent execution.

## Architecture Details

Slack Socket Mode event intake and Web API reply delivery are local to this
crate. The crate normalizes direct messages, app mentions, channels, and
threads into connector-core conversation keys.

Thread support is architectural, not cosmetic: `thread_ts` participates in
conversation identity so parallel Slack threads do not collapse into one Puffer
session.

## Contract Details

- Required config: bot token and Socket Mode app token.
- Optional config: allowed Slack users, mention policy, group-key policy, and
  welcome text.
- Mention contract: channel messages require app mention by default; direct
  messages do not.
- Thread contract: replies to threaded events must include the originating
  `thread_ts`; unthreaded replies go to the channel.
- Identity contract: allowed users compare Slack user ID strings.
- Output contract: messages must respect Slack length limits and be split before
  Web API delivery.
- Safety contract: bot/self messages are ignored.
