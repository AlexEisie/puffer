# puffer-connector-webhook

## Purpose

`puffer-connector-webhook` exposes a simple HTTP webhook that forwards JSON
messages into Puffer Code sessions and returns agent replies.

## Design

The connector is an Axum server. It uses the same connector-core runtime as chat
connectors but does not depend on a third-party messaging SDK.

Config covers bind address, request path, optional bearer auth token, allowed
origins, welcome behavior, and whether long responses should be split.

## Logic

HTTP handling must:

- Serve `GET /health` with a lightweight health response.
- Accept `POST <path>` JSON containing `conversation_id`, `message`, and
  optional `user_id`.
- Enforce `Authorization: Bearer <token>` when `auth_token` is configured.
- Enforce configured CORS origins when present.
- Treat webhook conversations as direct-message-like and per-chat.
- Run connector-core built-in commands before agent turns.
- Return structured success or error responses.

## Contract

The inbound JSON contract is intentionally small:

- `conversation_id`: required stable external conversation key
- `message`: required user text
- `user_id`: optional caller identity, defaulting to a webhook user

The response contract is JSON containing the generated reply or an explicit
error. Authentication failures must be HTTP authorization errors, not agent
messages.

Webhook delivery is synchronous from the caller perspective. The connector must
not assume the caller can receive out-of-band follow-up messages.

## Architecture Details

The webhook connector is a stateless HTTP transport around connector-core. It
does not maintain an external platform cursor or gateway connection. Each HTTP
request supplies the conversation identity needed for session continuity.

Axum routing, bearer-auth extraction, CORS policy, request validation, and JSON
response shaping are local to this crate.

## Contract Details

- Required config: bind address can default, but the endpoint must have a path.
- Default path: `/puffer`.
- Health contract: `GET /health` must return a lightweight success response
  without running an agent turn.
- Auth contract: when `auth_token` is configured, missing or wrong bearer token
  is rejected before message handling.
- Input validation: empty `conversation_id` or empty `message` is ignored or
  rejected before dispatch.
- Conversation contract: webhook messages are per-chat/direct by default.
- Output contract: HTTP response includes the full reply or explicit error; long
  response splitting is optional and controlled by config.
