# Datadog Webhook Normalization

## Change

The generic webhook connector now recognizes Datadog monitor and event webhook
payloads and converts them into `InboundMessage` values.

## Recognition

A payload is considered Datadog-shaped when one of these conditions is true:

- the payload or headers include an explicit Datadog source hint and the payload
  has a title, message, or alert status
- the payload has Datadog alert fields such as alert transition, alert cycle
  key, or alert id plus alert details
- the event type is one of Datadog's documented webhook event types and the
  payload has a title or message
- the payload links back to a Datadog app URL and has a readable notification
  title or message

Generic event payloads without Datadog-specific fields are intentionally
rejected.

## Conversation Routing

Conversation ids prefer stable Datadog identifiers:

- `datadog:alert_cycle:<alert-cycle-key>`
- `datadog:alert:<alert-id>`
- `datadog:incident:<incident-id>`
- `datadog:event:<event-id>`
- `datadog:event:<title>`

## Message Summary

Summaries include title, transition, status, priority, host, scope, metric,
query, message body, tags, logs sample, snapshot URL, and Datadog link when
present.

## Compatibility

Datadog webhooks support user-defined payloads. Puffer documents a recommended
custom payload but also accepts common flat field names so existing webhook
configurations can be routed without requiring an exact JSON envelope.
