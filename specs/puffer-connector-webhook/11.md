# Grafana Alerting Webhook Normalization

## Change

The generic webhook connector now recognizes Grafana Alerting webhook
notifications and converts them into `InboundMessage` values before falling
back to later provider-specific normalizers.

## Recognition

A payload is considered Grafana-shaped when it has a non-empty `alerts` array,
top-level `status` of `firing` or `resolved`, and at least one Grafana-specific
hint: `X-Grafana-Alerting-Signature`, `state` of `alerting` or `ok`, or an
`orgId` value. This avoids claiming generic Alertmanager payloads that share
the `alerts` and `status` shape without Grafana identity fields.

## Conversation Routing

Conversation ids are stable by receiver and group or alert subject:

- `grafana:<receiver>:group:<group-key>`
- `grafana:<receiver>:alert:<fingerprint>`
- `grafana:<receiver>:alert:<alertname>`

If no subject id exists, the connector falls back to a receiver-scoped Grafana
group conversation.

## Message Summary

Summaries include status, receiver, alert count, state, group key, common
labels, title, notification message, up to five alert summaries, source URLs,
truncated alert count, and the Grafana external URL when those fields are
present.

## Security

The normalizer records the Grafana signature header only as a shape hint.
Signature verification is intentionally not implemented yet because the
deterministic setup flow does not collect the Grafana webhook signing secret.
