# Azure DevOps Webhook Normalizer

The webhook connector now recognizes Azure DevOps Service Hooks payloads and
converts code, pull request, and work item events into provider-specific
inbound messages before the generic webhook fallback.

## Design

- Payload detection requires a known Azure DevOps event family such as `git.*`,
  `workitem.*`, or `ms.vss-code.*`, plus a service-hook-shaped `resource`
  object.
- Supported event families include Git pushes, pull request lifecycle events,
  pull request comments, and work item created, updated, and commented events.
- Conversation ids prefer stable identifiers in this order:
  `azure-devops:<project>:pull-request:<id>`,
  `azure-devops:<project>:work-item:<id>`,
  `azure-devops:<project>:push:<repository>:<branch-or-reference>`, then
  `azure-devops:<project>:<event>:<delivery-or-event>`.
- Message summaries include event type, project, actor, repository, pull
  request title, branch names, work item type, state, URL, detailed message
  text, and short commit summaries when present.

## Compatibility

The shape guard rejects unrelated event payloads that do not use Azure DevOps
code or work-item service-hook event identifiers. Existing provider
normalizers keep their precedence because Azure DevOps runs after Asana and
before Bitbucket.
