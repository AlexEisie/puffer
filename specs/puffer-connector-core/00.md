# puffer-connector-core

## Purpose

`puffer-connector-core` provides the shared runtime and contracts used by all
platform connectors. It turns inbound platform messages into Puffer sessions,
runs agent turns through `puffer-core`, and turns responses back into
platform-safe message chunks.

## Design

The crate deliberately keeps platform SDKs out of the core path. Platform
connectors provide `InboundMessage` values and configuration; the core runtime
handles session mapping, built-in connector commands, Puffer state hydration,
agent execution, response splitting, and retry helpers.

Main parts:

- `ConnectorRuntime` owns shared Puffer dependencies and dispatches messages.
- `ConnectorsConfig` loads raw connector TOML config.
- `ConversationKey` and `ConversationSessionMap` map external conversations to
  Puffer session IDs.
- `MessageSplitter` applies platform message-size limits without breaking code
  fences when possible.
- Built-in connector commands provide `/start`, `/new`, `/reset`, `/help`,
  `/status`, and `/usage`.

## Logic

Dispatch logic must:

1. Build a `ConversationKey` from platform, conversation, user, group state, and
   group-key policy.
2. Resolve an existing Puffer session or create a new one.
3. Rehydrate `AppState` from the session store.
4. Append the inbound user message to the session.
5. Execute an agent turn through `puffer-core`.
6. Persist assistant output, state snapshots, and reflection traces.
7. Return a `DispatchOutcome` to the platform connector.

Built-in slash commands are handled before the agent turn. Unknown slash
commands return no built-in outcome so the normal agent path can decide what to
do.

Session maps must be saved atomically through a temporary file and rename.

## Contract

`InboundMessage` is the platform-independent input contract. It carries the
conversation ID, user ID, text, optional thread ID, group/mention metadata, and
bot-self metadata.

`DispatchOutcome` is the platform-independent output contract. It can be a
plain reply, an agent reply with session ID and creation flag, or an ignored
message.

Group session policy must be explicit:

- `PerChat` maps the whole chat to one session.
- `PerUser` maps each user inside a group chat to a separate session.

Connector config must accept both root TOML shape and `[connectors]` shape.
Disabled connectors must not start. Unknown connector config is preserved as
raw JSON for the platform crate to parse.

The core crate must not depend on Telegram, Discord, Slack, Matrix, email, or
HTTP SDK types.

## Architecture Details

Connector-core is a platform adapter kernel. Platform crates own transport SDKs
and message delivery; this crate owns Puffer session routing and dispatch. The
runtime boundary is `ConnectorRuntime::dispatch`, which accepts a normalized
message and returns a normalized outcome.

State is split deliberately:

- Platform state stays in the platform crate or remote service.
- Conversation-to-session mapping lives in `ConversationSessionMap`.
- Durable transcripts and metadata live in `puffer-session-store`.
- Agent state is reconstructed into `puffer-core::AppState` per dispatch.
- Shared config/resources/providers/auth are held by the runtime and reused.

This architecture lets every connector share command behavior, session
continuity, message splitting, retry behavior, and agent execution without
sharing platform SDK code.

## Contract Details

- Input contract: `InboundMessage` text must be already filtered for platform
  message type but not yet interpreted as a Puffer command.
- Command contract: known connector slash commands are handled locally; unknown
  slash commands fall through to the agent path.
- Session contract: one `ConversationKey` maps to at most one active Puffer
  session until reset/remove.
- Persistence contract: session maps are pretty JSON and must be saved through
  temp-file rename.
- Reply contract: platform crates must run returned text through the correct
  `MessageSplitter` before sending.
- Error contract: dispatch failures return connector errors; platform send
  failures are retried by platform crates, not hidden in core.
- Isolation contract: no platform crate should need to know `AppState`
  restoration details.
