# Puffer Workflow

subscription / subscriber shall be part of workflow, renamed to connector / connection
A connector is a general connector like Telegram, Slack, etc.
A connection is an authorized connection like Telegram Login, Slack Login
A workflow is a [connection + filter] (trigger) to a subsequent action: any tool call (inheriting creator’s ACL, or self defined ACL by user) or triage an agent to handle or connector action, can be a graph, e.g., first run command from message then save to sqlite via bash tool, then send back a success

subscribe => delivers steaam of Output JSON, shall handle retry etc internally. For cases like user computer dead and restart, it shall provide the incremental changes from last pushed item, not delta after user computer restarts. Use stdout.

permission definition for each action slug

Connector as a proxy of agent
User can make a connector directly control an agent or multiple agents
For example, user can deploy a TG bot to control agent 1,2,3,4 or all agents they have
/connect <agent slug>, then send message and receive updates form the agent.
Underlyingly, Puffer defines a AgentProxy interface, which implementations define the connect phase and agent control phase.
For telegram bot for example:
Subscribing to messages
Connect phase: filter for command (/start, /connect …), on /start send how to connect, on /connect, toggle the agent proxy (persistent state) on for the agent, storing user => agent id.
Agent phase: If agent proxy relation exists, each message gets send to the agent, the response is then morphed to TG bot supported format and calling the connector’s act.
 Make it into a trait. For initial support, just do TelegramBot and SlackBot. Not all connector can be supported as a proxy of the agent.
