# puffer-config

## Purpose

`puffer-config` discovers Puffer configuration locations, loads layered TOML
configuration, exposes typed config values, and provides the settings catalog
used by CLI, TUI, daemon, and command handlers.

## Design

The crate centers on three concepts:

- `ConfigPaths` describes the workspace root, workspace `.puffer` directory,
  user home config directory, bundled resource directory, and related storage
  locations.
- `PufferConfig` is the typed runtime configuration structure.
- The settings catalog describes command-editable settings, their aliases,
  scopes, and value types.

Configuration is split by scope. User config stores personal defaults such as
model, provider, theme, editor mode, fast mode, effort, and copy behavior.
Workspace config can override workspace-operational values such as provider
base URLs, headers, and query parameters.

## Logic

Path discovery must:

- Use the process cwd as the workspace root unless an explicit workspace is
  passed.
- Use `.puffer` under the workspace for workspace config.
- Use `PUFFER_HOME` when set, otherwise the user home `.puffer` directory.
- Resolve bundled resources from `PUFFER_BUILTIN_RESOURCES_DIR` when set,
  otherwise from the workspace `resources` directory.

Config loading must:

1. Start from defaults.
2. Merge user TOML if present.
3. Remember user-selected model/provider/theme/editor/fast/effort/copy values.
4. Merge workspace TOML if present.
5. Restore those remembered user-selection values so workspace config does not
   silently replace personal preferences.
6. Apply fallback statusline data from `~/.claude/settings.json` only when a
   Puffer statusline is not already configured.

Saving must write only the requested scope and create parent directories as
needed.

## Contract

The typed config contract includes:

- Provider defaults: `default_provider`, `default_model`
- OpenAI transport overrides: `openai_base_url`, `openai_headers`,
  `openai_query_params`
- UI preferences: `theme`, `editor_mode`, `fast_mode`, `effort_level`,
  `copy_full_response`, `mascot`, and `ui`
- UI subfields: `no_alt_screen`, `tmux_golden_mode`, and `status_line`

Settings commands must accept the catalog aliases, including camelCase aliases
such as `editorMode`, `fastMode`, `copyFullResponse`, `effortLevel`,
`openaiBaseUrl`, `openaiHeaders`, and `statusLineCommand`.

Value parsing must be typed. Boolean settings accept booleans, nullable string
settings accept null-like values, string maps parse key/value data, and unsigned
integer settings reject negative values.

Missing config files are not errors. Invalid TOML, invalid setting names, and
invalid setting values are explicit user-facing errors.

## Architecture Details

The crate is a pure configuration layer. It does not know about terminal
rendering, providers beyond generic config fields, sessions, resources beyond
paths, or command execution. Callers supply a workspace root and receive typed
paths and typed config values.

Configuration architecture is layered:

- Built-in defaults are the base.
- User config represents personal preference and auth-adjacent defaults.
- Workspace config represents repo-specific behavior.
- Session-only settings are applied by higher layers and are not persisted
  through normal user/workspace saves.

The settings catalog is the single source of truth for command-settable config
keys, accepted aliases, value kinds, and persistence scope.

## Contract Details

- File format: TOML for user and workspace config.
- Missing files: treated as empty config.
- Invalid TOML: returned as an error with path context.
- Merge invariant: workspace config can override operational config but must
  not silently replace remembered user selections for model/provider/theme,
  editor mode, fast mode, effort, or copy behavior.
- Path invariant: `ConfigPaths` must be enough for callers to find workspace
  config, user config, resources, auth, sessions, and generated workspace files.
- Key aliases: camelCase aliases must round-trip to canonical snake_case fields
  through the settings catalog.
- Value parsing: CLI values must be converted before assignment so booleans,
  nulls, maps, strings, and unsigned integers cannot be stored in the wrong
  shape.
- Persistence: save helpers must create parent directories and write only the
  requested scope.
