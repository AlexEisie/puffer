# puffer-connector-email

## Purpose

`puffer-connector-email` connects an IMAP inbox and SMTP sender to Puffer Code
sessions. It supports email conversations where inbound mail becomes a Puffer
turn and responses are sent as outbound email.

## Design

The crate uses IMAP for polling and SMTP for replies, while all agent execution
goes through `puffer-connector-core`. Email-specific parsing, threading, sender
filtering, and delivery live in this crate.

Config includes IMAP and SMTP endpoints, credentials, sender address, optional
allowed senders, polling interval, group-key policy, and welcome behavior.

## Logic

Inbound handling must:

- Poll the configured IMAP inbox.
- Use email threading headers such as `References` and `In-Reply-To` when
  available to build stable conversation IDs.
- Ignore messages sent from the configured outbound address.
- Enforce `allowed_senders` case-insensitively when configured.
- Treat email as direct-message-like; mention gating is off by default.
- Dispatch to connector-core built-ins before agent execution.

Outbound handling must:

- Send replies through the configured SMTP relay.
- Preserve thread context when enough header data is available.
- Split very long responses with the email splitter limit.
- Retry transient SMTP failures.

## Contract

The connector requires usable IMAP and SMTP configuration before it can start.
Passwords are supplied by connector config; this crate does not own OAuth or
secret-manager flows.

Allowed sender checks are exact, case-insensitive mailbox comparisons in this
connector. Domain suffix matching belongs to the email subscriber helper, not
this connector.

The connector must not emit telemetry or retain mail bodies outside the Puffer
session and connector state needed for polling/threading.

## Architecture Details

The email connector has two adapters around connector-core: an IMAP poller and
an SMTP sender. The poller converts mailbox changes into inbound messages. The
sender converts connector outcomes into email replies.

Thread identity is derived from email headers where possible, with message
sender as user identity. Agent execution and transcript persistence remain
inside connector-core and session-store.

## Contract Details

- Required config: IMAP host/port, SMTP host/port, username/password, and
  outbound from address.
- Optional config: allowed senders, polling interval, group policy, and welcome
  text.
- Sender filtering: exact case-insensitive mailbox matching in this connector.
- Self-filtering: messages from the outbound address are ignored.
- Conversation contract: `References`/`In-Reply-To` should keep a thread mapped
  to the same session when available.
- Output contract: SMTP replies must use configured sender identity and preserve
  thread headers when enough metadata exists.
- Storage contract: durable conversation content belongs in Puffer sessions,
  not ad hoc mail archives.
