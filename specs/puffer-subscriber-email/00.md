# puffer-subscriber-email

## Purpose

`puffer-subscriber-email` is the hidden email subscriber process launched by
Puffer subscription runtime. It polls an IMAP mailbox, emits NDJSON events, and
can send email replies through SMTP control commands.

## Design

The process is a stdio subscriber. Stdout is reserved for NDJSON event output;
stderr is used for tracing and diagnostics. Persistent state lives in
`PUFFER_SKILL_STATE_DIR`.

State files include plaintext email config, a seen-UID high-water file, and any
runtime helper state required by the poller.

## Logic

Startup must:

- Resolve state dir from `PUFFER_SKILL_STATE_DIR`, defaulting to `./state`.
- Resolve topic from `PUFFER_SKILL_TOPIC`, defaulting to `email`.
- Load email config from state.
- Emit `config_required` when config is missing.
- Wait for an `EmailConfigure` command before polling if needed.

Polling must:

- Connect to IMAP over TLS.
- Select INBOX.
- On first successful run, set a high-water UID without emitting old backlog.
- On later runs, fetch messages newer than the high-water UID.
- Parse sender, subject, body, and message metadata.
- Ignore mail sent by the configured outbound address.
- Enforce allowed senders by exact address or `@domain` suffix.
- Emit `message` events with dedup keys and payload metadata.
- Persist high-water state atomically.
- Back off on polling errors.

SMTP command handling must send configured replies or emit control errors when
configuration is missing or invalid.

## Contract

The process reads `SubscriberCommand` JSON lines on stdin and writes `Event`
JSON lines on stdout. Non-JSON stdout is a protocol violation.

Supported control commands include email configuration and send-message
requests. Unknown commands emit control diagnostics instead of crashing.

Email passwords are stored in plaintext in the subscriber state directory. The
caller is responsible for filesystem permissions and secret handling policy.

Events must include topic, kind, optional dedup key, text, and payload so
`puffer-subscriptions` can route them.

## Architecture Details

The email subscriber is a child-process integration, not the chat email
connector. It emits events for subscription routing and accepts stdin control
commands. It owns its local polling state and SMTP command behavior but does
not own Puffer sessions or agent turns.

State files are intentionally simple so the process can be supervised,
restarted, and upgraded independently of the main Puffer process.

## Contract Details

- Environment contract: `PUFFER_SKILL_STATE_DIR` controls state location and
  `PUFFER_SKILL_TOPIC` controls emitted event topic.
- Stdout contract: every stdout line must be valid event JSON.
- Stderr contract: diagnostics and tracing only; no protocol data.
- Config contract: config is persisted TOML and must validate host/user/from
  fields before polling or sending.
- IMAP contract: first successful poll establishes baseline; later polls emit
  only new UIDs.
- Dedup contract: emitted messages include stable dedup keys based on mailbox
  data where available.
- SMTP contract: send-message commands produce control success/error events and
  must not pretend delivery succeeded before SMTP success.
- Secret contract: plaintext password storage is explicit and must be documented
  to callers through setup flows.
