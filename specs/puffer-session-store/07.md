# puffer-session-store Update 07: Chat Attachment Sidecars

## Summary

Session history can now reference durable chat attachments through structured
`StoredAttachment` records on `TranscriptEvent::UserMessage`.

## Contract

- `UserMessage.attachments` is an ordered list and defaults to empty when old
  transcript lines omit the field.
- Attachments are staged under
  `<sessions>/<session-id>.attachments/<attachment-id>/`.
- `original` contains the staged bytes and `metadata.json` contains the stored
  attachment record.
- Attachment IDs are generated UUID strings scoped to a session.
- Loading staged attachments rejects non-UUID IDs and metadata whose `id` does
  not match the requested ID.
- Attachment availability is computed from the presence of `original`; it is
  not persisted in the transcript.
- Deleting a session removes both existing session files and the matching
  attachment sidecar directory.

## Non-Goals

- No placeholder-text parser.
- No cross-session deduplication.
- No background garbage collection.
- No cloud or remote sync.
- No public API that exposes absolute attachment paths to desktop clients.

## Verification

Session-store tests should cover serialization, legacy deserialization,
staging, availability, missing-file behavior, invalid ID rejection, metadata ID
mismatch rejection, and session deletion cleanup.
