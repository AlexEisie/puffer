# puffer-session-store

## Purpose

`puffer-session-store` persists Puffer sessions, metadata, transcript events,
state snapshots, trace sidecars, listing, lookup, rename, tagging, and forking.

## Design

The session model is append-only. Each session has a metadata JSON file and a
JSONL transcript log. Runtime traces are stored as sidecar JSONL files with
sanitized trace names.

The metadata file contains user-facing and lookup fields. The transcript log
contains chronological `TranscriptEvent` values that can be replayed by
`puffer-core` to rebuild `AppState`.

## Logic

Session creation must allocate a UUID, initialize metadata, create the metadata
stub, and prepare the transcript path.

Event appending must:

- Serialize one JSON event per line.
- Update session `updated_at_ms`.
- Leave existing events untouched.
- Support explicit transcript rewrite events rather than rewriting history for
  clear/pop behavior.

Listing must load metadata, count transcript events, and sort newest-first by
`updated_at_ms`.

Lookup must support full UUID, UUID prefix, display name, and generated title.

Forking must create a new session with copied transcript events, copied tags and
note, and `parent_session_id` pointing to the source session.

## Contract

`SessionMetadata` fields are stable and migration-sensitive:

- `id`
- `display_name`
- `generated_title`
- `cwd`
- `created_at_ms`
- `updated_at_ms`
- `parent_session_id`
- `slug`
- `tags`
- `note`

Transcript event variants include user, assistant, system, tool invocation,
command invocation, rename, git diff snapshot, transcript rewrite, and state
snapshot events.

`TRACE_RUNTIME` is the runtime trace sidecar name. Trace appends must not block
normal transcript appends except for filesystem errors.

The store must not replace append-only history with opaque blobs. Future
migrations need to preserve event-level replay.

## Architecture Details

The session store is the durable transcript boundary. It stores session metadata
separately from event logs so list/search operations do not require replaying
every transcript. Event logs are append-only JSONL so future migrations can
process events incrementally.

Trace sidecars are intentionally separate from user-visible transcript events.
They can record runtime/reflection detail without changing the transcript
replay contract.

## Contract Details

- File contract: metadata lives in `<id>.session.json`; transcript events live
  in `<id>.jsonl`; trace sidecars use sanitized trace names.
- Append contract: appending an event must not rewrite previous event lines.
- Touch contract: appends and metadata mutations update `updated_at_ms`.
- Rewrite contract: clear/pop behavior is represented by `TranscriptRewritten`
  events, not by editing prior transcript files.
- Snapshot contract: state snapshots are optional replay accelerators and
  behavior records; replay must tolerate older snapshots missing newer fields.
- Lookup contract: session find supports UUID, UUID prefix, display name,
  generated title, and slug-like user labels where implemented.
- Fork contract: forked sessions preserve source transcript and metadata context
  while recording parent session ID.
- Failure contract: filesystem errors include path context and should not be
  swallowed by callers.
