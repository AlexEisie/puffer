# puffer-transport-anthropic

## Purpose

`puffer-transport-anthropic` implements Anthropic-compatible authentication,
request construction, fingerprinting, attribution blocks, CCH finalization,
response parsing, and usage helpers.

Anthropic compatibility is stricter than generic provider behavior and must be
preserved in this crate.

## Design

The crate exposes a curated transport API. It separates:

- API key, OAuth bearer, and session-ingress auth modes.
- OAuth URL, exchange, refresh, role fetch, and API-key creation helpers.
- Anthropic Messages request construction.
- Claude-compatible header ordering and user-agent behavior.
- Attribution block generation and CCH placeholder replacement.
- Response parsing for text and tool-use blocks.

Provider-neutral code should call this crate rather than reconstruct Anthropic
HTTP details elsewhere.

## Logic

Auth logic must support:

- API key auth through `x-api-key`.
- OAuth bearer auth through `Authorization: Bearer`.
- Session-ingress auth from environment variables, file descriptors, or token
  files, including organization UUID handling.
- Claude.ai inference OAuth scope detection.
- PKCE authorization, pasted-code parsing, token exchange, and refresh.

Request building must:

- Target `/v1/messages?beta=true`.
- Preserve modeled header order.
- Use a Claude-style `claude-cli/<version> (...)` user agent.
- Include session, Stainless, remote-control, client-app, workload, beta,
  dangerous-browser-access, Anthropic version, app, and request ID headers.
- Add auth headers appropriate to auth mode.
- Insert the attribution block as a standalone first system block.
- Compute fingerprint metadata and replace the first `cch=00000` placeholder
  with a deterministic CCH hash.

Response parsing must preserve known text and tool-use blocks while retaining
unknown block data where possible.

## Contract

`AnthropicAuth` is the auth contract: none, API key, OAuth bearer, or
session-ingress.

`BuiltAnthropicRequest` is the request contract: URL, ordered headers, JSON
body, and the attribution prefix block used to build that body.

The attribution block contract is strict. It must be standalone, first in the
system block list, and include Claude-compatible metadata such as version,
entrypoint, workload, fingerprint, and CCH value.

Do not simplify Anthropic into generic provider plumbing if doing so changes
header order, user agent, attribution block placement, fingerprinting, CCH, or
session-ingress behavior.

## Architecture Details

The Anthropic transport is a compatibility-critical wire-format crate. It owns
Anthropic-specific request shape, auth header shape, beta header composition,
Claude-style attribution, CCH, fingerprinting, and response parsing. Higher
layers should pass typed config into this crate rather than reconstructing those
details.

The request builder is deliberately more specific than generic HTTP helpers
because header order and system-block placement are part of the compatibility
surface.

## Contract Details

- Auth contract: API key, OAuth bearer, and session-ingress modes map to
  different headers/cookies and must remain distinct.
- OAuth contract: Console and Claude.ai auth flows use PKCE and scope-aware
  inference behavior.
- Session-ingress contract: token/env/fd/file discovery and organization UUID
  handling must be preserved for remote/session use.
- Header contract: modeled header order must remain stable where the repo
  represents it.
- User-agent contract: requests use Claude-style `claude-cli/<version> (...)`.
- Attribution contract: attribution is a standalone first system block.
- CCH contract: the first `cch=00000` placeholder is replaced with deterministic
  hash output during request finalization.
- Beta contract: beta header content depends on auth mode, model capability,
  prompt caching, context management, and interleaved thinking behavior.
- Parser contract: unknown response blocks are preserved enough for forward
  compatibility instead of discarded silently.
