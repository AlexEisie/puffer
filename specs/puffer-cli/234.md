# Monitor Field Rule RPCs

## Design

`task_monitor_rule_add` remains backward compatible with legacy keyword
requests while accepting explicit rule kinds:

- `kind: "keyword"` with `keywords` compiles literal regex matching over
  `Event.text`.
- `kind: "field"` with `field`, `operator`, and optional `value` loads the
  selected monitor connection schema and compiles the rule to `FilterSpec::Jq`.

The daemon locates schemas through the selected connection subscriber manifest
directory first. If no subscriber schema exists, it falls back to
`connectors/<connector_slug>/event_schema.json` under workspace, user, then
bundled resources. Desktop does not hardcode connector-specific fields.

## Keyword Operators

Keyword rules honor the optional `operator` parameter for `Event.text`.
Missing `operator` remains backward-compatible and defaults to `contains`.
`contains` compiles escaped literal alternatives, `equals` compiles whole-text
anchored alternatives, and `matches` validates and stores user regex patterns.
`exists` is rejected for message text rules.

Field rules are unchanged: they require connector event schema metadata and
compile through `compile_event_field_rule`.

Connections without monitor rule schema metadata still accept keyword rules,
because those target `Event.text`, but field rules are rejected instead of
guessing connector payload fields.

## Persistence

Field and keyword rules share the existing monitor persistence model. Include
rules update `include_filter` / `include_filters`; exclude rules append to
`ignore_filters`; deletion still removes the exact displayed rule JSON.

## Snapshot Contract

`workflow_list` attaches `monitor_rule_schema` to monitor-capable connections
and monitor bindings when a schema exists. A missing schema is represented by
`null`, and clients should still allow keyword rules.

Snapshots expose the latest `include_filters` and `ignore_filters` arrays after
rule add/delete calls so Desktop can rebuild readable condition chips when the
settings view is reopened.
