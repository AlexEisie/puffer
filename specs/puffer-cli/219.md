# Inferred Contact Store Split

## Summary

Inferred contact proposals now persist outside the saved-contact store. Saved
contacts remain in `.puffer/runtime/contacts.json`, while unsaved inferred
contacts live in `.puffer/runtime/inferred_contacts.json`.

## Design

- `contacts.json` is the canonical store for user-curated `SavedContact` rows.
- `inferred_contacts.json` stores the last inferred `ContactProposal` rows.
- `contacts_list`, `contacts_search`, and `contacts_infer` still return
  `proposals` in the RPC snapshot, so the desktop UI contract is unchanged.
- Saving a contact prunes overlapping inferred proposals from the inferred
  store without treating them as saved contacts.
- Legacy `contacts.json.proposals` data is sanitized, pruned against saved
  contacts, and migrated into `inferred_contacts.json` before new writes omit
  proposal data from `contacts.json`.

## Compatibility

Existing workspaces that already wrote proposals into `contacts.json` keep
those proposals on the next list, infer, or save path. New contact-store writes
do not serialize proposal rows into the saved-contact file.
