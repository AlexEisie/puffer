# Windows OAuth Browser Open

## Summary

CLI and daemon-host OAuth login no longer open authorization URLs through a
hand-built `cmd /C start` command. The shared authflow helper now delegates to
the existing `webbrowser` crate, avoiding Windows shell splitting of OAuth query
parameters separated by `&`.

## Behavior

- `puffer auth login`, automatic OAuth start paths, and the desktop daemon RPC
  keep using the same `open_browser(url) -> bool` contract.
- The authorization URL is passed directly to `webbrowser::open` instead of
  being embedded in a Windows `cmd.exe` command line.
- If the browser cannot be launched, callers still fall back to their existing
  error or manual-copy messaging.

## Compatibility

OAuth bundle creation, PKCE verifier/state handling, callback parsing, token
exchange, and credential persistence are unchanged. This only changes the OS
browser-launch mechanism for the already-generated authorization URL.
