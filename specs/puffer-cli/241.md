# Desktop Turn Stream Order

Desktop turns must preserve assistant text deltas and tool activity in the same
chronological order seen during the live stream. Text emitted before a tool call
is stored before that tool invocation; text emitted after a completed tool call
is stored after it.

Tool-call requests reserve their chronological slot in the turn progress ledger.
When the matching invocation completes, it fills that reserved slot. This keeps
completed tools from jumping ahead of assistant text that streamed after the
request but before the tool result arrived.

On successful completion, the daemon appends ordered assistant text and completed
tool invocations from the turn progress ledger instead of storing all tools
before the final assistant text. If the final assistant text contains a suffix
not already emitted as text deltas, that suffix is appended as the final
assistant message.

On cancellation, the daemon persists the user prompt if needed, ordered assistant
text segments, completed tool invocations, synthetic failed invocations for
pending tool calls, the `Interrupted by user.` system message, and the finished
turn boundary. This keeps interrupted progress visible without stacking all text
into a single final-looking answer around the activity panel.

When session history is converted to desktop timeline items, assistant messages
emitted inside an active turn boundary keep that turn id. This lets the desktop
preserve one visual assistant response for the turn while still deciding which
items are normal prose and which items are operational activity.

The desktop timeline treats `SendUserMessage` and `Brief` invocations as
presentation messages. Their JSON `message` input is emitted as an assistant
message with the active turn id, while operational tools such as `Sleep`,
Canvas, and filesystem or browser calls remain tool activity rows.
