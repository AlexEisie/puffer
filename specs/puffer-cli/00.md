# puffer-cli

## Purpose

`puffer-cli` is the top-level binary crate for Puffer Code. It owns the
process entrypoints for interactive use, auth management, session commands,
daemon mode, desktop integration, connector startup, workflow commands, and
internal subscriber helpers.

The crate is the boundary between user-facing process behavior and the shared
libraries that implement config, resources, providers, sessions, tools,
workflows, subscriptions, and the TUI.

## Design

The binary is intentionally thin around shared crates:

- CLI parsing and command visibility live in `cli_args` and `command_surface`.
- Startup loads `puffer-config`, `puffer-resources`,
  `puffer-provider-registry`, auth storage, and the session store once, then
  passes owned handles to the selected command path.
- Interactive sessions enter `puffer-tui`.
- Hidden JSON and daemon entrypoints expose automation surfaces without making
  them normal user commands.
- Long-lived process modes can start workflow, subscription, and connector
  runtimes; one-shot commands avoid those background services.

The daemon is a WebSocket RPC server with shared state. The shared state owns
config, loaded resources, the provider registry, auth store, session store,
in-flight turns, UI state, browser registries, PTY registries, file watches,
and broadcast event channels.

## Logic

Startup logic must:

1. Parse the requested CLI command.
2. Discover config paths from the current workspace and `PUFFER_HOME`.
3. Load layered config and resources.
4. Load provider descriptors and auth credentials.
5. Decide whether the process is one-shot or long-lived.
6. Dispatch to the requested subcommand, daemon, desktop API, connector, or TUI
   path.

Daemon logic must:

- Require the caller to present the daemon token in the WebSocket URL.
- Send a handshake containing the URL, token, protocol version, and workspace
  root.
- Accept correlated JSON-RPC-like requests with `id`, `method`, and `params`.
- Return either a correlated result or error.
- Publish events for session, UI, terminal, browser, workflow, file, and agent
  state changes.
- Route agent-turn permission requests through the daemon permission flow before
  continuing tool execution.

Connector startup loads `.puffer/connectors.toml` first and then
`~/.puffer/connectors.toml`, honors disabled connectors, and starts only
connectors compiled into the current build.

## Contract

Public command behavior is defined by the `puffer` executable. Stable user
commands include auth, session, provider, resource, plugin, workflow, MCP, and
interactive entrypoints. Hidden commands such as `desktop-api`, `daemon`,
`serve`, `__subscriber`, and request-fixture helpers are internal automation
interfaces and may be narrower than the public CLI.

Auth commands must preserve the supported surface:

- `puffer auth status`
- `puffer auth set-api-key <provider> [--stdin]`
- `puffer auth clear <provider>`
- `puffer auth oauth-url <provider>`
- `puffer auth oauth-start <provider>`
- `puffer auth oauth-exchange <provider> --verifier ... [--state ...] [--stdin]`
- `puffer auth oauth-refresh <provider>`
- `puffer auth login <provider> [value] [--stdin]`

The daemon protocol contract is token-authenticated WebSocket JSON. Request
handlers must return structured errors instead of panicking and must not expose
filesystem, terminal, browser, auth, or provider operations without checking the
daemon session and method-specific rules.

The desktop API contract is newline-delimited JSON over stdin/stdout for
SSH-backed desktop integration. It returns explicit success or error JSON for
session, repository, auth, settings, and workflow operations.

This crate must not introduce telemetry, analytics, feedback upload flows, or
Claude-branded marketing commands.

## Architecture Details

The crate has five architectural surfaces:

- Public CLI: argument parsing, help visibility, user-facing auth/session/plugin
  commands, and interactive launch.
- Hidden automation CLI: daemon, desktop API, request fixtures, subscriber
  helpers, and benchmark runner.
- Long-lived services: daemon WebSocket server, connector runtimes,
  subscription runtime, and workflow runtime.
- Process integration: terminal detection, stdin/stdout JSON modes, auth file
  mutation, and signal-safe shutdown paths.
- Dependency assembly: config paths, resources, provider registry, auth store,
  and session store are loaded here and passed into downstream crates.

`puffer-cli` should not implement provider turns, TUI rendering, tool execution,
resource parsing, workflow validation, or subscription routing directly. It
constructs those systems and translates process-level input/output around them.

## Contract Details

- Config inputs: cwd, `PUFFER_HOME`, resource overrides, CLI flags, stdin,
  connector TOML, auth JSON, and optional daemon token material.
- Runtime outputs: terminal UI, command stdout/stderr, daemon WebSocket frames,
  desktop API JSON responses, connector replies, and session-store mutations.
- Auth mutation: commands that write credentials must use
  `puffer-provider-registry::AuthStore` and must save after mutation.
- Daemon RPC: every request must be correlated by ID; events are uncorrelated
  broadcasts; auth/config/session/file/browser/terminal/workflow methods must
  return structured errors.
- Desktop API: input and output must remain machine-readable JSON with explicit
  error objects; human prose belongs in values, not framing.
- Background services: startup failures should degrade to explicit command
  errors or warnings; shutdown must drain owned runtimes where handles exist.
- Exclusions: no telemetry, no analytics, no feedback upload, no Claude
  product-marketing commands, and no direct Tauri runtime dependency.
