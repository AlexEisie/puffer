# puffer-cli update 39

Workflow CLI import flags now use connection terminology while preserving old
subscription flag aliases.

`puffer workflow register` accepts:

- `--connection-slug` with alias `--subscription-topic`
- `--connection-pattern` with alias `--subscription-pattern`
- `--classify-prompt`

The CLI still imports native workflow JSON through the historical
`workflow register` subcommand for operator compatibility. Model-visible
workflow creation is handled by the `WorkflowCreate` resource in
`puffer-core` and persists direct connection workflow bindings.

Startup autostart logic reads workflow binding `connection_slug` fields instead
of the removed `source_topic` field and starts only matching legacy subscriber
manifests when those are still in use.

Runtime wiring:

- Startup installs a connector action executor so `connector_act` workflow
  actions and `ConnectorAct` tool calls run connector-defined `act` commands
  when available, including permission metadata in the result summary.
- Built-in message-send actions can still fall back to legacy subscriber
  commands for bridges that have not moved to the full connector protocol.
- Startup installs a workflow action runner that implements native workflow
  runs, direct tool calls, and agent triage actions for connection-triggered
  workflows.
- A background auth monitor runs routine `auth-ok` checks, refreshes active
  connection consumers, and prints a one-time repair notice when a connection
  stops authenticating.
