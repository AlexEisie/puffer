# Native Image Attachment Hydration

Desktop uploaded images now enter model requests through native multimodal
input instead of temp-path text hints.

## Daemon Contract

- Image attachments are preserved as durable attachment metadata on
  `AppState` before runtime dispatch.
- Image attachments are hydrated to transient `data:` URLs immediately before
  the provider request is built.
- Hydrated image payloads stay in memory only and are not written to session
  JSONL.
- Image attachments are no longer materialized under `/tmp/puffer-attachments`
  for model input.

## File Attachments

File attachments may still be materialized to temp paths so models can use
`Read` against local files. `build_model_input` emits path hints only for file
attachments and leaves image-only user text unchanged.

## Validation

Hydration validates image MIME type, image count, per-image byte size, total
raw-image bytes, original file existence, and size drift before provider
dispatch. Validation is two-phase so a failing turn does not partially hydrate
later images.

## Failure Behavior

If hydration fails, the daemon publishes an `attachment-hydration` turn error,
removes the in-flight turn, and returns before persisting an unsendable user
message.
