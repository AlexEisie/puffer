# puffer-tui

## Purpose

`puffer-tui` is the native Ratatui/Crossterm interactive UI for Puffer Code. It
renders transcripts, input, status, slash-command popups, overlays, permission
prompts, tool details, streaming output, and startup flows.

## Design

The TUI is transcript-first. `TuiState` owns interactive UI state while
`puffer-core::AppState` owns session behavior. Rendering is split into modules
for transcript layout, markdown, overlays, popups, status areas, top panels,
and tool detail.

The public entrypoint is `run_app`, with `StartupAction` controlling normal
startup versus resume picker startup.

## Logic

`run_app` must:

1. Determine whether an interactive terminal is available.
2. Support non-interactive single-submit mode when an initial prompt is given.
3. Configure alternate-screen or inline mode according to config.
4. Initialize `TuiState`.
5. Handle startup overlays and queued prompts.
6. Loop through render, input polling, pending provider-turn polling, and
   background-task wakeups.
7. Restore terminal state on shutdown.

Input logic must support UTF-8 cursor movement and editing, slash popup
selection, shell shortcut input, Ctrl+C double-press behavior, scroll/follow
state, pending submissions, queued prompts, loop mode, and permission prompts.

Turn execution is run in a background thread and streamed into UI events such
as thinking deltas, text deltas, tool-call requests, tool invocations,
reflection checkpoints, retries, usage, permission requests, and finish events.

Overlay logic maps slash commands to list, status, help, config, context,
doctor, permission, skill, plugin, MCP, IDE, hook, sandbox, session, rewind,
memory, tag, onboarding, and usage views.

## Contract

`run_app` must leave the terminal usable after normal exit or errors.

The TUI must dispatch commands through `puffer-core` so slash-command behavior
matches daemon, connector, and CLI automation surfaces.

Rendering must remain deterministic enough for terminal snapshot tests and tmux
parity tests. Markdown rendering must use the shared markdown renderer rather
than raw model text when possible.

Permission prompts must block only the relevant pending tool decision and must
return allow, deny, or session-level allow-all decisions to `puffer-core`.

## Architecture Details

The TUI is a frontend over `puffer-core::AppState`, not a separate application
engine. It owns terminal interaction, render state, overlays, and user input
editing while delegating commands, provider turns, permissions, and transcript
state to core.

Runtime work is split between the UI loop and background turn execution. The UI
loop remains responsive by polling pending events and redrawing; provider/tool
progress is streamed back through typed pending-submit events.

## Contract Details

- Entry contract: `run_app` accepts config/resources/providers/auth/session
  dependencies and either starts interactive mode or single-submit mode.
- Terminal contract: alternate screen, inline/no-alt-screen, cursor visibility,
  raw mode, and cleanup are restored on exit paths.
- Input contract: editing must preserve UTF-8 correctness and stable cursor
  positioning.
- Command contract: slash commands are sourced from `puffer-core` command
  metadata plus resource-backed skills.
- Render contract: transcript, statusline, overlays, permission prompts, and
  tool details should be derived from state rather than duplicated side data.
- Streaming contract: thinking/text/tool/usage/retry/reflection/permission
  events update visible UI without corrupting transcript order.
- Snapshot contract: tmux and terminal snapshot tests depend on deterministic
  golden-mode rendering.
- Error contract: provider and command errors should become visible transcript
  or status messages while leaving the terminal usable.
