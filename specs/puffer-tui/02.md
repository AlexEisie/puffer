# Browser permission prompt wording parity

## Purpose

Keep the TUI permission overlay aligned with the current `puffer-core` Browser
grant semantics so Browser-bearing prompts do not imply a wider session grant
than the runtime currently records.

## Design

`ApprovalOverlay` currently branches on `PermissionPromptRequest.browser`.

Under the current runtime behavior, generic prompts offer:

- once
- tool-for-session
- all-tools-for-session
- deny

Browser prompts instead offer:

- once
- browser-for-session
- deny

The Browser-specific second option is a presentation choice for the current
runtime contract. It still maps onto the existing `AllowSession` prompt
action, which `puffer-core` currently resolves into the appropriate Browser
scoped grant.

## Logic

- While Browser prompts remain scoped in `puffer-core`, the TUI should not
  render an `AllowAllSession` option row for them.
- While that same runtime behavior remains in place, Browser prompts should
  not describe the `a` shortcut as a tool-wide session grant.
- Browser prompts should still accept uppercase `A` as a case-insensitive
  shortcut for the session-scoped Browser option.
- Non-Browser prompts keep the existing `A` shortcut for `AllowAllSession`.

## Contracts

- As long as Browser `allow_all_session` continues to resolve to a Browser
  scoped grant, TUI wording should not claim it unlocks all tools for the
  session.
- Browser prompt copy should stay aligned with the active desktop/frontend
  wording unless the products intentionally diverge.
- Prompt rendering and keyboard shortcuts should stay deterministic for
  snapshot and unit tests.
