# puffer-workflow

## Purpose

`puffer-workflow` implements native Puffer workflow definitions, validation,
storage, cron triggering, DAG execution, and run history.

## Design

Workflows are file-backed definitions stored under workspace `.puffer/workflows`.
The crate supports Puffer workflow envelopes and raw AgentFlow-like pipeline
documents when registration options provide the missing trigger metadata.

Main parts:

- `WorkflowDefinition` describes slug, enabled state, trigger, and pipeline.
- `WorkflowTrigger` supports cron and subscription triggers.
- `Pipeline` and `Node` describe a DAG of agent steps.
- `DagRunner` executes nodes through a caller-supplied `AgentExecutor`.
- `WorkflowStore` persists definitions and run records.
- Cron helpers parse five-field local cron expressions and dedupe minute
  firings.

## Logic

Validation must reject:

- Empty or invalid slugs.
- Missing triggers.
- Empty node lists.
- Duplicate node IDs.
- Unknown dependencies.
- Dependency cycles.
- Unsupported node types.
- Missing prompts.
- Unsupported AgentFlow keys such as optimizer, remote, fanout, shell, Python,
  targets, sync, and graph optimization fields.

DAG execution must:

1. Mark nodes pending.
2. Run ready nodes whose dependencies completed.
3. Interpolate trigger and dependency output values into prompts.
4. Record completed or failed node results.
5. Skip downstream nodes when dependencies fail.
6. Append a run record to the store.

Store writes must use mutex-protected snapshots and atomic temp-file renames.
Run listing must return newest runs first.

## Contract

The workflow schema default is `puffer.workflow.v1`.

Supported triggers are cron and subscription. Cron uses five local-time fields.
Subscription triggers may include source-topic matching, optional event pattern,
and optional classification prompt.

Supported node type is agent-style execution. Shell, Python, remote, fanout,
and optimizer execution are explicitly out of scope until implemented.

The runner contract is caller-supplied `AgentExecutor`, so this crate does not
own provider auth, TUI state, or command dispatch.

## Architecture Details

The workflow crate is definition, validation, scheduling, and DAG execution
infrastructure. It intentionally excludes arbitrary shell/Python/remote
execution until those node types have explicit contracts.

Storage is workspace-local and file-backed. Runtime execution is dependency
injected through `AgentExecutor`, which lets CLI/daemon/subscription callers
choose how an agent step is actually run.

## Contract Details

- Definition contract: workflow documents must validate before being stored or
  run.
- Slug contract: slugs are stable workflow identifiers and must be safe for
  file/list/display usage.
- Trigger contract: cron and subscription triggers are the only supported
  trigger kinds.
- Cron contract: five-field local cron expressions are parsed with list, range,
  step, and Sunday handling; dedupe prevents repeated firing for one minute.
- DAG contract: dependencies must reference known nodes and must be acyclic.
- Node contract: only agent-style nodes are executable; unsupported keys are
  rejected up front.
- Interpolation contract: prompts may reference trigger and prior node output in
  the supported token syntax.
- Run contract: each run records node statuses, outputs/errors, timestamps, and
  skipped downstream nodes.
- Store contract: definitions and runs are written through atomic snapshot
  updates and listed in deterministic order.
