# puffer-subscriptions update 02

Subscriptions are now implemented as connection-driven workflow bindings.
Legacy `SubscriptionSpec` and `SubscriptionStore` names remain aliases while
the persisted schema uses workflow binding terminology.

Core model:

- `ConnectorTemplate` describes reusable connector capabilities, auth
  requirement, typed command argv, output schema, action schemas, and
  connector-defined permission metadata.
- `ConnectionRecord` describes one authorized connector instance with the
  lifecycle `created -> authenticating -> authenticated -> active -> degraded
  -> disabled`.
- `WorkflowBindingSpec` binds one `connection_slug` plus a Puffer-side filter
  to an action or action graph.
- `WorkflowHistoryStore` records direct workflow runs as trigger info plus
  action log entries.
- `AgentProxyStore` persists connector chat/user bindings created by
  `/connect <agent>` for proxy-capable connectors.

Runtime logic:

- Connections become `active` only when at least one enabled workflow binding
  or agent proxy consumer needs them; otherwise authenticated connections keep
  no stream running. Proxy-capable bot connectors count as consumers so their
  `/start` and `/connect <agent>` setup commands can be received before a
  user-specific binding exists.
- Connections act as a mux: multiple workflow bindings can consume the same
  connector event stream.
- Custom connector commands can run `auth-ok`, `act`, and `subscribe` through
  typed argv. Subscribe processes receive a JSON command on stdin, emit JSONL
  event/checkpoint/health frames on stdout, and receive ack frames only after
  the host processes all matching proxy/workflow actions and persists the
  cursor. If a matched action fails, the cursor remains unacked so the
  connector can retry from its last pushed item.
- Dead connector stream tasks are pruned during consumer refresh and restarted
  when the connection still has an enabled consumer.
- Routine auth checks mark broken connections `degraded` and set
  `auth_failure_notified` once so callers can surface a one-time user notice.
- Filters are evaluated inside Puffer, not the connector, using JSON subset,
  regex, or a small jq-like syntax with regex support.
- Actions support SQLite insert, connector action, tool call, agent triage,
  native workflow run, legacy forwarding, and dependency-ordered action graphs.
- Built-in TelegramBot and SlackBot proxy implementations handle `/start`,
  persist `/connect <agent>` bindings, route bound messages to agent targets,
  and render agent responses back into connector action payloads. Proxy
  bindings store both the external principal and the current reply target so a
  user can be bound to an agent while replies still go to the originating
  Telegram chat or Slack channel.
- Direct connector-stream processing runs the proxy/workflow router before ack;
  the older subscriber event bus remains available for legacy manifests.

Compatibility:

- Existing subscription JSON with `subscriptions`, `id`, `source_topic`, and
  `prefilter` still loads.
- Existing subscriber-runtime manifests still run for built-in Telegram user
  and Email flows.
- Built-in connector templates cover `telegram-login`, `telegram-bot`,
  `slack-bot`, and `email`; external command execution is available for custom
  connectors while built-ins continue using their existing in-process bridge
  until platform drivers expose the full protocol.
