# Monitor Event Schemas

## Design

Monitor rule field filters are driven by connector-owned `event_schema.json`
files. The schema describes payload-relative field paths, UI labels, primitive
types, allowed operators, and optional enum or boolean display values. Keyword
rules are not schema fields: they always match the unified `Event.text` that
subscribers emit.

## Contracts

- `version` must be `1`.
- Field paths are payload-relative dotted identifiers and may be nested.
- Duplicate paths, empty operators, unsupported operators, and enum fields
  without values are invalid.
- `compile_event_field_rule` only accepts fields declared by the schema.
- Field rules compile to existing `FilterSpec::Jq`; `equals` preserves JSON
  scalar types for boolean and number fields, and `exists` uses a non-empty
  presence predicate so null, empty strings, empty arrays, and empty objects do
  not match.
- Keyword rules continue to compile to `FilterSpec::Regex` against
  `Event.text`.

## Bundled Schemas

Bundled subscriber schemas cover `telegram-user`, `gmail-browser`, `email`, and
`gcal-browser`. Bundled command/connector schemas cover `telegram-bot`,
`lark-login`, and `lark-bot` under `resources/connectors/<slug>`. Gmail and
GCal use nested paths such as `message.subject` and `event.title`; Telegram,
Lark, and IMAP email mostly expose flat payload fields.
