# puffer-subscriptions update 03

Adds a Telegram-shaped end-to-end connector workflow test and hardens the
connector action path it exercises.

Design and behavior:

- Connector stream processing may invoke `connector_act` from inside the
  subscription runtime worker. `SubscriptionManager::run_connector_action` and
  `check_connection_auth` now bridge through `tokio::task::block_in_place`
  when already inside a runtime, avoiding nested `block_on` panics while
  preserving the synchronous manager API used by workflow tools.
- Workflow template rendering now supports dotted JSON payload paths, such as
  `{{payload.chat.id}}` and `{{payload.from.name}}`, while preserving existing
  one-level substitutions like `{{payload.from}}`.
- The Telegram E2E test registers a local Telegram-shaped connector, creates
  an authenticated connection and workflow binding, receives one connector
  stream event, saves it to SQLite, invokes a connector reply action, persists
  the cursor, and verifies the connector received an ack.

Compatibility:

- Existing template syntax remains valid.
- The synchronous manager API remains unchanged.
- The runtime bridge assumes the subscription manager is backed by the
  multi-thread Tokio runtime used by `puffer-cli`.
