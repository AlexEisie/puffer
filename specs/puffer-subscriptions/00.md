# puffer-subscriptions

## Purpose

`puffer-subscriptions` routes subscriber events to user-defined actions such as
SQLite inserts, outbound forwarding, or workflow runs.

## Design

The crate separates durable subscription specs, live subscriber event intake,
classification/prefiltering, and action dispatch.

Main parts:

- `SubscriptionSpec` describes one route from a source topic to an action.
- `SubscriptionStore` persists specs.
- `SubscriptionManager` owns subscribers, event bus, router, dispatcher, and
  classifier.
- `SubscriptionRouter` consumes events and applies matching specs.
- `BuiltinActionDispatcher` implements SQLite, forward-message, and workflow
  actions.

## Logic

Spec validation must reject:

- Empty IDs or topics.
- Invalid regex prefilters.
- Unsafe SQLite table identifiers.
- Empty forward targets.
- Empty workflow slugs.
- Unknown action variants.

Routing logic must:

1. Receive all events from the subscriber event bus.
2. Load current subscription specs.
3. Skip paused specs and topic mismatches.
4. Apply optional regex prefilter.
5. Apply optional classifier.
6. Dispatch the configured action.
7. Update seen, matched, acted, and failed stats.

SQLite actions must create a table with stable event columns and insert by
`envelope_id` with idempotent conflict handling. Forward actions must use an
installed outbound bridge. Workflow actions must use an installed workflow
runner and pass event data as trigger payload.

## Contract

`SubscriptionSpec` fields include ID, description, source topic, status,
prefilter, optional classification prompt/model, action, and creation time.

Supported actions are:

- `SqliteInsert { path, table }`
- `ForwardMessage { platform, target, template }`
- `RunWorkflow { slug }`

Templates support `{{text}}` and `{{payload.field}}` substitutions.

The router must keep running when one spec or action fails. Failures are counted
and surfaced through stats rather than crashing the manager.

## Architecture Details

Subscriptions sit between low-level subscriber events and high-level actions.
They do not poll external services directly and do not execute agent provider
turns directly unless the configured action routes to an installed workflow
runner.

The architecture has four replaceable pieces:

- Store: durable subscription definitions.
- Event source: subscriber runtime event bus.
- Classifier: optional event classifier for semantic filtering.
- Dispatcher: action executor for SQLite, forwarding, and workflows.

This split allows tests and callers to install null/custom classifiers,
outbound bridges, and workflow runners.

## Contract Details

- Store contract: specs are durable, validated, and reloadable while the router
  runs.
- Topic contract: events match specs by exact source topic before any expensive
  filtering.
- Prefilter contract: regex filters are compiled at validation and applied to
  event text with configured case behavior.
- Classifier contract: classifier rejection skips action dispatch without
  counting as an action failure.
- SQLite contract: table names are validated as safe identifiers; inserts are
  idempotent by envelope ID.
- Forward contract: no installed outbound bridge is an explicit dispatch error.
- Workflow contract: workflow action payloads include event metadata and must
  not drop original event payload JSON.
- Stats contract: seen, matched, acted, and failed counters are observable
  runtime state.
