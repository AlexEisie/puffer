# Monitor Triage Debounce

Monitor triage events from both the live event bus and command-backed connector
streams are now debounced per stable conversation scope before dispatch. The
router queues events only when the matching topic has an enabled `monitor-*`
triage binding and no immediate non-triage workflow binding, preserving
existing low-latency behavior for forwards, tool actions, and custom workflows.

The debounce key is derived from stable payload scope fields in priority order:
`thread_id`, `conversation_id`, `chat_id`, `channel_id`, then `room_id`, with
camelCase aliases accepted. Matching events are held for a short quiet period
and then passed through the existing batch processing path, so the normal
filters, classifiers, dedup handling, history writes, runtime guard injection,
and action accounting still apply.

Command-backed connector streams ask their installed processor for the same
debounce key and quiet-window delay before acknowledging events. This lets a
short Telegram or Lark burst become one processor batch instead of one triage
turn per message while preserving the existing ack-after-success contract.

This change is intentionally conservative. Events without a stable conversation
scope continue to process immediately, and topics with mixed workflow actions
are not delayed.
